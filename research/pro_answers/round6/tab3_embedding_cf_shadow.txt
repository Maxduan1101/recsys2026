1. 推荐路线

优先级如下：

尝试 LightGBM LambdaRank

最可能吃到非线性收益。

能自动利用：source rank interaction、intent gate、specificity、exact-match、seed similarity、popularity prior。

训练和推理代码更短。

但不要直接全量替换 BM25 排名。

同时保留 numpy/scipy pairwise ranker

作为无依赖 fallback。

适合做“保守排序校准器”。

线性模型可解释，容易加权重边界，比如强制 BM25/rule/exact-match 权重非负。

不要先做 pointwise logistic

pointwise logistic 可以当 baseline，但它优化的是“这个候选是不是 gold”，不是“同一 turn 内谁排前面”。

你这个数据是每个 group 一个或少数正例，pairwise 更自然。

2. 特征编码：重点是 rank 变换和 group 内标准化

每条 JSONL 应该至少有：

group_id = session_id + turn_id
track_id
label = 1 if gold else 0
legacy_rank
source_ranks
rrf_score
rule_boost
intent
category
specificity
track_priors

exact_match_features
per-source rank 特征

不要直接只喂 rank=37。每个 source 做这组：

source_present = 0/1
source_recip = 1 / (rank + 60)      # RRF style
source_inv_rank = 1 / rank
source_log_rank = -log1p(rank)
source_top1 = rank <= 1
source_top3 = rank <= 3
source_top5 = rank <= 5
source_top10 = rank <= 10
source_top20 = rank <= 20

missing rank 不要填 999 后直接训练。更稳的是：

source_present = 0
source_recip = 0
source_inv_rank = 0
source_log_rank = 0
top flags = 0

这样模型不会把“missing=999”学成奇怪的线性数值。

legacy anchor 特征

BM25 是已知强 anchor，所以专门强化：

legacy_present
legacy_rank
legacy_recip = 1 / (legacy_rank + 60)
legacy_top1/top3/top5/top10/top15/top20
legacy_score_z_in_group
legacy_rank_percentile

如果有多个 BM25 版本，例如 metadata_all, title_artist, train_context, goal_query，要保留：

max_recip_across_sources
source_consensus_top20_count

这些能让模型知道“一个 item 是单源偶然命中，还是多源一致命中”。

score 类特征

对这些做 group 内 z-score 或 percentile：

rule_boost
bm25_score
embedding_score
seed_similarity
popularity

因为不同 turn 的 BM25/RRF 分布不一样，原始分数跨 group 不可比。推荐同时保留：

可以用，但必须小心：

风险点：popularity 很容易把模型推回热门老歌，提升局部 dev 但伤 catalog diversity，也可能伤 broad intent。建议对 popularity 加上限，或者只允许在 tail rerank 中作为 tie-breaker。

3. 训练 / dev split

必须按 user/session 分组切分，不要按 row 随机切：

train sessions: 80%
dev sessions: 20%

原则：

同一个 session 的所有 turns 必须在同一 split
同一个 group_id 的所有 candidates 必须在同一 split
candidate generation 必须 train/dev 独立复现
dev 上不能 positive-inject 后再报主指标

如果你有 official train/dev：

train official train
tune official dev
blind-like validation = 从 train 内再切一个 session-level holdout

原因是你现在已经发现 local dev 和 blind 有明显 mismatch，所以不能只相信一个 dev 数。

4. 泄漏风险清单

最危险的是这些：

positive injection 泄漏
如果训练或 dev candidate pool 里强行注入 gold，可以用于诊断“reranker 上限”，但不能作为 honest end-to-end 指标。

track document augmentation 泄漏
只能用 train turns 给 track 文档加用户 query / assistant explanation / selected reason。不能用 dev gold，更不能用 blind 相关信息。

goal_progress 对齐错误
goal_progress_assessment[turn_number] 必须先确认它评价的是同 turn 的 music item，还是 transition 到下一个 turn。没确认前，只能把它作为弱反馈，不要强行当 positive/negative label。

future-history leakage
previously_recommended, user_feedback, session_seed 只能来自当前 turn 之前，不能看未来 turns。

track_id memorization
不要 one-hot track_id。可以用 smoothed train popularity，但不要让模型记住“这个 track 经常是答案”。

dev-tuned postprocess 过拟合
如果你对 dev 调了很多 alpha/K/gate，要再用一个 blind-like holdout 复核。

6. scipy/numpy pairwise ranker 方案

实现一个线性 RankNet 风格模型：

对每个 group，取 gold candidate p 和若干 negatives n，构造：

d = x_pos - x_neg
loss = log(1 + exp(-w · d)) + λ ||w||²

也就是希望：

w · x_pos > w · x_neg

负例采样优先取 hard negatives：

legacy top20 negatives
RRF top20 negatives
exact/source-consensus false positives
currently outranking gold negatives
random tail negatives

每个 positive 配 20–50 个 negatives 就够了。

推荐加入权重边界：

legacy_recip weight >= 0
rrf_score weight >= 0
rule_boost weight >= 0
exact_match weight >= 0
negative_feedback_match weight <= 0
previously_recommended weight <= 0
popularity weight in [-0.2, 0.2]

因为 scipy 的 L-BFGS-B 支持 bound-constrained optimization，所以这个很适合做“安全线性模型”。
SciPy 文档

推理时不要直接用：

final_score = model_score

而是用：

final_score = z_legacy + alpha * clip(z_model, -2, 2) + beta * exact_match_bonus

alpha 从小到大试：

0.05, 0.10, 0.15, 0.20, 0.30

只要 dev 上 top5/top10 开始掉，就不要再加。

7. 安全上线：legacy-head protection

我建议第一版提交级策略是：

K = 15
final[1:15] = legacy_bm25[1:15] unchanged
final[16:20] = model rerank from candidate pool excluding protected items

如果 dev 非常稳，再试：

K = 10
final[1:10] = legacy unchanged
final[11:20] = model rerank

不要一开始 rerank 全 top20。

允许进 head 的例外

只有这些情况允许模型把 item 提到 1–10：

title_exact_match and intent_exact_title
artist_exact_match and intent_artist
album_exact_match and intent_album
source_consensus_top20_count >= 3 and model_margin very high
negative_feedback demands replacing repeated/bad item

否则模型只能动 tail。

final ranking 公式

推荐：

base = legacy_recip_z
model = model_margin_z
exact = exact_entity_bonus
repeat = repeat_penalty
pop = clipped_popularity_prior

final_score =
    base
  + alpha * clip(model, -2, 2)
  + beta * exact
  - gamma * repeat
  + delta * pop

其中：

alpha = 0.05 ~ 0.20
beta = 0.2 ~ 0.8
gamma = 0.1 ~ 0.5
delta = 0.00 ~ 0.05

popularity 只做 tie-breaker，不要让它主导。

1. 使用的 retrieval channels

不要一次性把 7 个 embedding 全混进去。最小安全版本只用 4 个主通道：

Channel A：seed_metadata

用历史正向 seed tracks 的 metadata-qwen3 向量求均值。

query_vec = mean(L2(track_metadata_vec[positive_seed_tracks]))
score = dot(query_vec, track_metadata_vec[candidate])
topN = 200

作用：补 BM25 在自然语言语义上的同类歌曲召回。

Channel B：seed_attributes

用历史正向 seed tracks 的 attributes-qwen3 向量求均值。

query_vec = mean(L2(track_attributes_vec[positive_seed_tracks]))
score = dot(query_vec, track_attributes_vec[candidate])
topN = 200

作用：补 mood / genre / vibe / activity 类请求。

Channel C：seed_track_cf

用历史正向 seed tracks 的 track cf-bpr 向量求均值。

query_vec = mean(L2(track_cf_vec[positive_seed_tracks]))
score = dot(query_vec, track_cf_vec[candidate])
topN = 200

作用：协同过滤式“喜欢这个，也可能喜欢那个”。

Channel D：user_cf

如果当前 session / user 有 user cf-bpr：

score = dot(user_cf_vec[user], track_cf_vec[candidate])
topN = 200

作用：个性化 prior。

3. seed track 提取规则

只用当前 turn 之前的信息。

positive_seed_tracks =
    previous music tracks in same session
    excluding tracks with explicit negative feedback
    excluding tracks after DOES_NOT_MOVE_TOWARD_GOAL if alignment is verified

如果 goal_progress_assessment 对齐还没完全确认，就不要强依赖它。更安全的第一版：

positive_seed_tracks = previous recommended / played tracks before current turn

5. channel score normalization

每个 turn 内，每个 channel 单独归一化：

z_channel_score =
    (score - mean(topN_scores)) / std(topN_scores)

missing channel 记为 0，并加 present flag：

metadata_present
attributes_present
track_cf_present
user_cf_present

不要直接混 raw dot product，因为不同 embedding 通道的分布不可比。

6. 最小融合公式

第一版不用训练模型，直接 deterministic scoring：

embedding_score =
    0.35 * z_metadata
  + 0.25 * z_attributes
  + 0.25 * z_track_cf
  + 0.15 * z_user_cf

如果没有 seed，只剩 user_cf：

embedding_score = z_user_cf

最终 tail score：

tail_score =
    0.60 * legacy_tail_score_z
  + 0.30 * embedding_score
  + 0.10 * rule_boost_z
  - 0.20 * repeat_penalty

其中：

legacy_tail_score_z = BM25/RRF group-normalized score
repeat_penalty = same track / same artist overused in batch or session

这不是为了最强，而是为了安全。BM25 仍然主导。

8. hard gates：什么候选允许进 16–20

embedding 候选必须满足至少一个条件：

appears in legacy top100
or appears in at least 2 embedding channels top200
or user_cf rank <= 50
or seed_cf rank <= 50
or metadata/attributes rank <= 30

否则不准进最终 top20。

再加 negative filters：

if track was already recommended in same session: block
if artist explicitly rejected: block
if exact negative feedback match: block
if candidate has no metadata/id alignment check: block

这个能防止单通道偶然高分污染尾部。

embedding incremental oracle recall@200 < 1% of groups
changed groups 中超过 55% 是负向或无效替换
catalog_diversity 没提升，nDCG 还掉

更直接一点：

如果 embedding 只能提高 unique_tracks，但不能带来 net_hits >= 0，
那它不是 ranking improvement，只是 diversity perturbation。

这种情况下不要用 embedding 改 ranking，最多用于 response explanation 或未来 LTR features。

Step 2：写 topK retrieval function
Python

def topk_by_dot(query_vec, matrix, k=200):
    scores = matrix @ query_vec
    idx = np.argpartition(-scores, k)[:k]
    idx = idx[np.argsort(-scores[idx])]
    return idx, scores[idx]

13. 最推荐的最终实验矩阵

只跑 6 个就够：

B0: legacy unchanged

S0: shadow only, no ranking change

E1: protect top18, channels = metadata + attributes + track_cf
E2: protect top18, channels = metadata + attributes + track_cf + user_cf

E3: protect top15, channels = metadata + attributes + track_cf
E4: protect top15, channels = metadata + attributes + track_cf + user_cf

判断逻辑：

如果 E2/E4 比 E1/E3 差，user_cf 太 noisy，关掉。
如果 E1/E2 都没收益，embedding 不适合直接 rerank。
如果 protect top18 稳但 top15 不稳，只用 top18。
如果 shadow oracle 有收益但 actual rerank 没收益，说明需要 LTR，不适合规则融合。
14. 最终建议

第一轮不要证明“embedding 很强”，而是证明：

embedding/CF 在 BM25 top15 之后有没有安全边际收益。

我会这样落地：

默认提交候选：
BM25 top15 frozen
16–20 从 BM25 top100 ∪ metadata top200 ∪ attributes top200 ∪ track_cf top200 ∪ user_cf top200 中选择
必须通过 2-channel / top50 / legacy-top100 gate
tail_score = 0.60 legacy + 0.30 embedding + 0.10 rule - repeat_penalty

最保守版本：

BM25 top18 frozen
只替换 19–20
只在 positive_seed_count >= 1 的 turns 启用

如果这个实验不能满足：

nDCG 几乎不掉
net_hits >= 0
catalog diversity 上升
has-seed subset 有收益

那就conclusively reject embedding/CF as a direct ranking component for this submission cycle，保留它们作为未来 LTR features，而不是现在强行融合。

Likely fixes for your exact setup
Fix 1: Replace uniform RRF with legacy-anchored, group-capped RRF

Uniform RRF is treating metadata_all, title_artist, and album_artist as three independent confirmations, even though they are mostly correlated field views of the same evidence. Collapse them into one metadata group vote.

A good first production candidate:

1. Retrieve:
   legacy top200
   metadata/title/album top100 each
   tags top200
   train-context top200
   goal top100
   seed/history top100

2. Score with group-capped weighted RRF.

3. Hard rules:
   - Keep legacy top5 fixed.
   - Positions 6–10: allow an auxiliary item only if it is also in legacy top100
     or appears in at least 2 source groups.
   - Positions 11–20: allow at most 3 aux-only items.

This gives the new sources room to recover missing positives without letting them erase the high-confidence legacy prefix.

Fix 2: Use all sources for candidate generation, then rerank by calibrated features

Fix 3: Make source authority conditional

Use gates, not global weights.

For conversation_goal:

gate_goal = 1.0 for turns 1–2
gate_goal = 0.5 for turns 3–5
gate_goal = 0.2 for turns 6–8
gate_goal *= 0.5 if latest user message contains correction/negation
gate_goal *= 1.5 if latest query is very short or vague

For seed_history:

gate_seed = 1.0 only for positive-feedback history
gate_seed = 0.3 for neutral prior recommendations
gate_seed = 0.0 or negative penalty after rejection/negation

For title_artist and album_artist:

boost only when named entities are present:
- artist name in query/history
- track title-like quoted phrase
- album mention
- "from the album", "by", "song called", "that track"

For tags and train-context docs:

never promote above rank 10 unless:
- legacy also retrieves the item in top100, or
- at least two independent groups agree, or
- exact artist/entity match exists

Fix 5: Rebuild train-context-enriched item docs as low-noise descriptors

Train-context item docs can help if they contain phrases that explain why a track was recommended. They hurt if they memorize synthetic assistant phrasing, include wrong-turn context, or overrepresent popular tracks.

Build them like this:

For each ground-truth track in train:
  collect only user utterances before that track was recommended
  collect positive assistant rationale only if the turn moved toward the goal
  exclude turns marked or inferred as negative
  strip UUIDs
  strip generic phrases:
    "here are", "you might like", "since you enjoyed"
  keep artist, album, genre, mood, era, instrumentation, energy words
  cap per-track context length

Then treat this source as:

candidate recall: yes
top-10 authority: mostly no
Fix 6: Do diversity after relevance ranking, not inside top-rank fusion

Since catalog diversity and lexical diversity are computed separately from nDCG, do not buy diversity by damaging positions 1–10. 
GitHub
 Use a two-zone list:

positions 1–10: relevance locked
positions 11–20: mild diversity rerank

For positions 11–20, apply MMR or a simple penalty:

score_tail =
  relevance_score
- 0.02 * same_artist_already_in_list
- 0.01 * same_album_already_in_list
+ 0.01 * catalog_novelty

Only apply this when relevance scores are close, for example within 2–3% of each other. Lexical diversity should mostly be handled in predicted_response, not by changing tracks.

The decision tree I would use

After running the diagnostics, choose fixes based on this table:

Observation	Meaning	Fix
Union oracle barely beats legacy	New sources do not find extra positives	Use them only for tail diversity
Union oracle beats legacy, RRF loses	Ranking/fusion failure	Use weighted group-capped RRF or reranker
One source has high unique Hit@200 but poor nDCG@20	Good recall, bad precision	Candidate-only source
Source helps only certain specificity/category segments	Conditional value	Add gates by goal/query type
Full top20 protection recovers nDCG	Top-rank dilution	Protect top5/top10, cap aux-only items
Tags improve diversity but lose hits	Noisy semantic expansion	Clean tags and restrict to positions 11–20
Seed/history hurts later turns	False-positive seed drift	Sentiment/feedback gate previous music IDs

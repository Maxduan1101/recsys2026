4. 决策指标：不要被 catalog diversity 误导

你给的 Blind A 设置是：

80 rows × top20 = 1600 recommendation slots
catalog size = 47071
max possible catalog_diversity = 1600 / 47071 ≈ 0.0340

你现在：

head20 unique tracks ≈ 1216
catalog ≈ 1216 / 47071 ≈ 0.0258

head15/head18 tail diversity 这种方案如果把 unique tracks 提到 1348：

catalog ≈ 1348 / 47071 ≈ 0.0286
absolute gain ≈ +0.0028

按你推测的 composite：

composite ≈ 0.5*nDCG
          + 0.1*catalog
          + 0.1*lexical
          + 0.3*((judge-1)/4)

那么 catalog +0.0028 只贡献：

0.1 * 0.0028 = +0.00028 composite

这意味着只要 nDCG 掉：

0.5 * ΔnDCG < -0.00028
=> ΔnDCG < -0.00056

catalog gain 就被抵消了。

所以：tail diversity 只能在几乎零 nDCG 损失时提交。 你之前看到 dev nDCG 从 0.08587 掉到 0.08180，哪怕这个全量 dev 不 blind-like，也已经说明它不是天然安全方案。

6. Paired bootstrap and tests

The test must be paired because candidate and anchor are evaluated on the same rows.

6.1 Panel-level paired delta

For each panel p:

Δ_p = metric(candidate, panel_p) - metric(anchor, panel_p)

Then report:

mean(Δ_p)
median(Δ_p)
p05(Δ_p)
p10(Δ_p)
p25(Δ_p)
p75(Δ_p)
p90(Δ_p)

This is the most intuitive output.

6.2 Bootstrap confidence interval

Use bootstrap over panels:

Python
运行
for b in range(5000):
    sampled_panels = sample_with_replacement(primary_panels)
    boot_delta[b] = mean(delta[p] for p in sampled_panels)

ci_low = percentile(boot_delta, 5)
ci_high = percentile(boot_delta, 95)

Report one-sided lower confidence bound:

LCB_90 = percentile(boot_delta, 10)
LCB_95 = percentile(boot_delta, 5)

For risky ranking changes, require:

LCB_90(delta_ndcg) >= 0

For tail-only diversity changes, you may allow tiny negative tolerance:

LCB_90(delta_ndcg) >= -0.001

But only if hard safety gates pass.

6.3 Row-level paired sign test

For row-level nDCG deltas:

wins   = count(delta_ndcg > epsilon)
losses = count(delta_ndcg < -epsilon)
ties   = rest

Use epsilon = 1e-9 or 1e-4.

A candidate should not have many more loss rows than win rows unless the losses are only tiny tail movements.

Simple gate:

loss_rows <= win_rows * 1.25

For LTR/full rerank, use stricter:

loss_rows <= win_rows
6.4 Stratified bootstrap

For the final report, also run a stratum-preserving bootstrap:

For each bootstrap sample:
  for each Blind-A stratum:
      sample quota[stratum] rows with replacement
  compute paired delta

This avoids a bootstrap sample accidentally drifting away from Blind A’s distribution.

HARD FAIL  -> reject immediately
SOFT FAIL  -> warn, manual review only
PASS       -> candidate can replace anchor
7.1 Format gates

Immediate reject:

- missing row
- invalid track_id
- fewer than 20 recommendations
- duplicate track_id inside one top20
- response missing
- response contains obvious placeholder
- submission row order mismatch
7.2 Ranking identity gates for response-only commits

If commit is labeled response_only, require:

top20 list identical to anchor for 100% rows
rank order identical to anchor for 100% rows

If not identical:

REJECT

Response-only commits should never accidentally change ranking.

8. Diversity and response gates

Because catalog diversity has small weight relative to nDCG, never accept a ranking-damaging change for a tiny diversity gain.

For tail-diversity commits, require:

catalog_diversity absolute gain >= +0.0015
unique_tracks gain >= +5%
artist_repeat_rate decreases

But also:

estimated composite LCB_90 > 0

Use your approximate composite:

Δcomposite =
    0.5 * ΔnDCG
  + 0.1 * Δcatalog
  + 0.1 * Δlexical
  + 0.3 * Δjudge_scaled

For local validation, Δjudge_scaled is usually unknown. So compute two versions:

conservative_composite:
  assume Δjudge_scaled = 0

optimistic_response_composite:
  use local proxy for response quality

Auto-accept should use conservative composite.

Response gates:

distinct_2 increases by at least +20% relative

hallucinated_artist_or_title_rate <= 1%

1. Upgrade from fixed 500 panels to posterior shadow-blind panels

Your current method likely does:

Blind A has observed stratum counts:
  turn/category/specificity -> count_s

Sample 500 local panels with the same count_s.

That is good, but still too brittle because Blind A has only 80 rows. The observed stratum mix is noisy. Instead, use:

Blind A observed distribution = one noisy observation
true blind distribution       = latent distribution
local panels                  = samples from possible true blind distributions

Use a Dirichlet-smoothed posterior over strata:

stratum s = turn_bucket × category × specificity

n_s = number of Blind A rows in stratum s
q_s = background distribution from local dev candidate pool
α   = smoothing strength, e.g. 10 to 30

θ ~ Dirichlet(n_s + α q_s)

For each panel:
    sample θ
    convert 80 * θ_s into integer quotas
    sample local rows by those quotas

This gives two uncertainty sources:

1. row uncertainty:
   Which dev rows represent Blind-like cases?

2. distribution uncertainty:
   What if Blind B / private is not exactly the same 80-row mix as Blind A?

This is stronger than fixed stratified panels because it does not over-trust the exact public 80-row composition.

2. Use macro-strata for sampling, micro-strata for diagnostics

Do not make the sampling grid too sparse. Use this as the main quota stratum:

macro_stratum =
    turn_bucket × category × specificity

Where:

turn_bucket:
  early = 1-2
  mid   = 3-5
  late  = 6-8

category:
  exact_track
  artist_or_album
  genre_style
  mood_activity
  era_language_region
  similar_to_seed
  feedback_refinement
  broad_exploration
  continuation_or_other

specificity:
  S3 = exact entity
  S2 = multi-constraint
  S1 = one broad meaningful constraint
  S0 = weak / ambiguous / continuation

Then add micro-diagnostics, but do not force exact quotas on them unless you have enough data:

micro features:
  exact title entity present
  exact artist entity present
  negative feedback present
  positive seed track present
  history length bucket
  query length bucket
  constraint count
  anchor BM25 confidence bucket
  anchor top20 diversity bucket

Important: the sampler may use only input-side or anchor-side frozen features. It must not use candidate performance, candidate rank of gold, or gold difficulty.

3. Split dev into three roles

Do not repeatedly inspect the same 8000-row dev pool and keep tuning to it. Make a fixed split by session:

dev_train_or_modeling:
  used for LTR fitting, source weights, prompt/template iteration

dev_selection:
  used for routine 80-row shadow panels

dev_lockbox:
  used rarely, only for pre-submit final check

Recommended split:

60% modeling
25% selection
15% lockbox

All split decisions should be by session_id, not by row, so that one conversation cannot leak across training and validation.

Rules:

- The 500-panel daily report uses dev_selection only.
- The full pre-submit report uses dev_selection + dev_lockbox, but reports lockbox separately.
- You do not inspect worst-row examples from lockbox unless the candidate is already rejected.
- You never tune thresholds after looking at lockbox.

This is the simplest way to avoid dev-overfitting.

5. Pairwise confidence interval method

For close submission packages, do not compare each package to dev average. Compare package A and package B on the same panels.

For each panel p:

Δ_p = metric(package_A, panel_p) - metric(package_B, panel_p)

Then report:

mean Δ
median Δ
p05 Δ
p10 Δ
p25 Δ
P(Δ > 0)
bootstrap LCB_90
bootstrap LCB_95

Use paired bootstrap over panels:

Python
运行
for b in range(5000):
    sample panel indices with replacement
    boot_delta[b] = mean(delta_p for sampled panels)

LCB_90 = percentile(boot_delta, 10)
LCB_95 = percentile(boot_delta, 5)

For final decisions, use the lower bound, not the mean.

Recommended rule:

A beats B only if:
  P(Δcomposite > 0) >= 0.90
  and LCB_90(Δcomposite) > 0
  and hard safety gates pass

For nDCG-specific ranking changes:

A beats B only if:
  P(ΔnDCG > 0) >= 0.90
  and LCB_90(ΔnDCG) >= 0

For tail-only diversity changes, allow a tiny nDCG tolerance only if diversity gain is meaningful:

LCB_90(ΔnDCG) >= -0.0005
median ΔnDCG >= 0
catalog gain >= +0.0015
no exact/S3 degradation
6. Use a transfer-error buffer

Because Blind A and local panels already disagreed once, add a transfer uncertainty penalty.

For each submitted package where you have public feedback:

local_predicted_delta = local shadow-blind delta vs anchor
public_observed_delta = Blind A delta vs anchor

transfer_error = public_observed_delta - local_predicted_delta

Maintain:

τ = max(
    abs(last_transfer_error),
    historical_RMSE_transfer_error,
    minimum_buffer
)

Use a minimum buffer such as:

τ_nDCG = 0.001 to 0.002
τ_composite = 0.0005 to 0.001

Then accept a close candidate only if:

LCB_90(local Δmetric) - τ > 0

This is very important. Without a transfer buffer, the evaluator will keep selecting tiny local wins that disappear on the public 80 rows.

Use these rules.

Rule 1: freeze the evaluator version

Every report should say:

evaluator_version = shadow_blind_v3

strata_schema_hash = ...
anchor_package = ...

Do not silently change strata, thresholds, or panel seeds after seeing a bad result.

Rule 2: separate “development panels” and “selection panels”

Use:

dev_visible_panels:
  allowed for debugging

dev_selection_panels:
  used for accept/reject

dev_lockbox_panels:
  used only before submission

You may inspect row-level failures in visible panels. For selection panels, inspect only grouped summaries. For lockbox, inspect only final pass/fail unless the package is rejected.

Rule 3: keep a commit budget

For example:

Only one full accept/reject report per serious candidate family:
  response-only
  head18-tail
  embedding-tail
  LTR-protected

Do not run 30 minor variants and pick the best p-value. That is leaderboard overfitting, just happening locally.

Rule 4: use family-level champion selection

Within each family, select using development panels:

best_response_only
best_head18_tail
best_embedding_tail
best_LTR_protected

This reduces multiple-comparisons damage.

10. Close-package decision policy

When packages are close, use this priority:

1. Does either package fail a hard gate?
2. Does either package have negative exact/S3 diagnostics?
3. Does either package have positive LCB_90 on nDCG or conservative composite?
4. Is the improvement larger than transfer buffer τ?
5. Is the gain from a high-weight metric or a low-weight metric?
6. Is the package simpler and more reproducible?

Recommended ranking of trust:

response-only ranking-identical package
> head18 tail-only package with flat nDCG
> embedding/LTR tail package with positive LCB
> head15/head10 diversity package
> full rerank package

For close packages:

If CI overlaps zero and hard gates pass:
    choose the safer one, not the higher mean.

If safer one is current public-safe anchor:
    keep anchor and save the candidate as backup.

If candidate only wins catalog diversity:
    do not submit unless nDCG is statistically flat.

12. Final recommended protocol

Use this as your automatic accept/reject flow:

1. Generate candidate submission.zip.

2. Run format + validity gates.

3. Classify package type:
   response_only / head18_tail / head15_tail / embedding_tail / LTR_protected / full_rerank.

4. Run posterior primary panels:
   1000 panels, 80 rows each.

5. Run fixed Blind-A-shaped panels:
   500 panels.

6. Run smoothed private-style panels:
   500 panels.

7. Run stress panels.

8. Compute paired deltas vs current champion:
   nDCG
   hit@20
   catalog
   lexical proxy
   conservative composite

9. Bootstrap paired panel deltas:
   report mean, median, p10, LCB_90, P(delta > 0).

10. Apply transfer-error buffer τ.

11. Apply hard safety gates.

12. Decision:
   ACCEPT only if LCB_90 - τ > 0 and no safety gate fails.
   WARN if mean positive but CI overlaps zero.
   REJECT if safety gate fails or LCB clearly negative.
   BACKUP if useful but not statistically submit-worthy.

My practical default for your current stage:

Submit only ranking-identical response improvements,
or ranking changes whose posterior-panel LCB_90 clears zero after transfer buffer.

For all close LTR/embedding/tail-diversity variants:
preserve submission.zip backups, do not submit,
unless they beat the anchor across posterior, fixed Blind-A, smoothed, and stress panels.

The key upgrade is this:

Do not ask: “Did this win on 500 sampled dev panels?”

Ask: “Does this still win after accounting for:
  paired uncertainty,
  Blind A distribution uncertainty,
  transfer error from public feedback,
  exact-entity safety,
  and repeated local tuning?”

That is the standard you need before trusting a close package under an 80-row public leaderboard.

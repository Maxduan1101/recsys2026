主人，lowest-risk answer: keep BM25 top-15 frozen first, only edit 16–20; then try top-10 frozen / edit 11–20 only if blind-like validation is consistently positive. Do not submit any package that lets multi-source RRF enter the head unless it passes very strict intent/entity gates.

The official evaluator makes this a good strategy: predictions are ordered, duplicates are invalid, and nDCG uses the discounted position formula, macro-averaged over turns and sessions. So moving or adding a correct item in positions 11–20 can help, while freezing top-10 protects nDCG@10 and most of the proven BM25 signal.

Core rule

Use BM25 as the anchor, not one feature among many.

final_top20 =
  BM25[1:K] frozen
  + guarded_rerank_or_replace(BM25[K+1:20], extra_candidates)

Run scopes in this order:

Scope A: K = 15, editable = 16–20   # submit-safe
Scope B: K = 10, editable = 11–20   # only if validation is robust
Scope C: K = 5,  editable = 6–20    # probably too risky; research only

For the current situation, Scope A is the first serious submission candidate.

Scoring candidates in the editable tail

Candidate pool per row:

C = original_BM25_tail
  ∪ exact_entity_candidates_topM
  ∪ gated_RRF_candidates_topM
  ∪ seed/history similarity candidates_topM
  ∪ popularity/train-prior candidates_topM

But a new candidate may enter only if it passes eligibility:

eligible(c) =
  c not in frozen_topK
  and c not duplicate
  and c valid track_id
  and evidence_count(c) >= threshold
  and not contradicted by negative feedback

Use a conservative rank-based score, not raw source scores:

score(c) =
  2.00 * legacy_bm25_rank_score(c)
+ 1.50 * exact_entity_score(c)
+ 0.70 * intent_source_score(c)
+ 0.40 * seed_positive_score(c)
+ 0.25 * popularity_prior(c)
+ 0.25 * train_prior(c)
+ 0.20 * original_tail_bonus(c)
- 1.00 * negative_feedback_penalty(c)
- 0.50 * generic_tag_penalty(c)

Where:

rank_score_s(c) = 1 / log2(rank_s(c) + 1), else 0

The key is original_tail_bonus: BM25 tail items should win ties. A replacement must beat the original candidate by a margin:

replace only if score(new) > score(weakest_original_tail_item) + τ

Suggested margins:

τ = 0.08 for positions 16–20
τ = 0.12 for positions 11–20
τ = 0.18 for any attempt touching 6–10

1. Baseline tail reorder only: no new tracks

First test:

freeze 1–15
rerank only original BM25 positions 16–20

Then:

freeze 1–10
rerank only original BM25 positions 11–20

No external candidates yet. This answers: “Is BM25’s own tail order suboptimal?”

Use:

score = BM25_score
      + exact_entity_bonus
      + positive_seed_bonus
      + train_prior_tiebreak

Expected result: tiny but very safe. If this loses, your entity parser/prior is hurting and should not be trusted for replacement.

2. Positions 16–20 exact-entity replacement

This is the first real improvement candidate.

freeze 1–15
editable 16–20
candidate_pool = BM25[16:20] ∪ exact_entity_candidates

Promote only high-confidence entity matches:

track title exact in last user utterance or goal
artist exact with artist-intent wording
album exact with album-intent wording
title + artist both match

Do not promote on weak matches:

single common word title
generic tag
negated entity: "not X", "tired of X", "avoid X"
history-only mention without current preference
noisy user tags: sleep, cds I own, to live by, etc.

Recommended promotion rule:

if title_exact and not negated:
    can enter 16–20
if artist_exact and candidate has BM25/RRF support:
    can enter 16–20
if album_exact and candidate has album/title/artist support:
    can enter 16–20
else:
    no replacement

This is the best low-risk nDCG move because exact entities are one of the few signals strong enough to beat the BM25 anchor without learning a full reranker.

3. Positions 16–20 gated RRF replacement

Now let RRF help only when it is not fighting BM25.

freeze 1–15
editable 16–20
candidate_pool = BM25[16:20] ∪ RRF_top50

Eligibility:

candidate must satisfy at least 2:
  appears in legacy BM25 top100
  appears in train-context BM25 top50
  appears in title/artist/album source top50
  positive seed similarity top50
  exact entity weak/medium match
  popularity/train prior above median

Reject if:

only tags source supports it
only train-context source supports it
source support comes from noisy tags
artist/album contradicts explicit request
negative feedback similarity is high

Use per-intent source gates:

Intent type	Allowed extra sources	Unsafe sources
Exact track	title/artist, metadata_all, exact entity	tags-only, popularity-only
Exact artist	title_artist, artist_album, metadata_all	mood tags-only
Album request	album_artist, metadata_all	generic train-context
Mood/activity	tags, metadata_all, attributes, popularity	exact title random matches
“more like this”	seed sim, CF/user sim, metadata_all	pure popularity
Negative feedback	sources dissimilar to rejected track	same artist/album/tag unless explicitly requested
Broad recommendation	popularity/train prior + BM25	aggressive RRF

This should be submitted only if the changed-row delta is positive and unchanged-row count remains high.

4. Popularity/train priors as tie-breakers, not drivers

Use popularity and train priors only after BM25/entity/source evidence. The challenge metadata includes popularity and release-date fields, and the dataset/evaluator setup is grounded in track metadata and user/context fields. 
GitHub

Good use:

candidate A and B have similar BM25/RRF/entity evidence
choose the one with stronger train prior or popularity

Bad use:

insert globally popular tracks into 16–20 with no query evidence

Suggested priors:

pop_prior = clipped_zscore(log1p(track_popularity))
train_prior = clipped_zscore(log1p(train_gold_count))
artist_prior = clipped_zscore(log1p(train_artist_gold_count))

Cap them hard:

pop_prior, train_prior ∈ [-1, +1]
combined prior contribution <= 0.40

Also test a turn-aware prior:

turn_prior(track, turn_number)
intent_prior(track, intent_type)

This is safer than global popularity because the public blind distribution may reward common “synthetic listener” patterns, but global head stuffing can easily collapse catalog diversity and hurt exact-intent rows.

1. Primary experiment: anchored weighted RRF

Current ensemble is probably close to this:

base_score(q, t) =
  Σ_m 0.25 / (K + rank_m(q, t))

Replace it with:

score(q, t) =
  Σ_m w_m / (K + rank_m(q, t))
  + λ / (K + base_rank(q, t))

Where:

m ∈ {four LTR models}
K ∈ {20, 40, 60, 80}
w_m ≥ 0
Σ w_m = 1
w_m ∈ [0.15, 0.40]
λ ∈ {0.00, 0.10, 0.20, 0.35, 0.50}

The λ / (K + base_rank) term is important. It says: the current 0.18348 ensemble is the anchor; new weights must overcome it, not replace it freely.

Candidate pool should be exactly the existing four-model union, for example:

U_q = union of top100 candidates from the four current LTR models

No seed-CF rescue. No train-session mixing. No new retrieval source.

Hard candidate guard:

A candidate may enter top20 only if:
  candidate is in current base top30
  OR candidate appears in at least 2 model top50 lists
  OR candidate appears in one model top5 list

This prevents one weird model from injecting single-source noise.

2. Objective function

Do not optimize raw nDCG only. Optimize an anchored objective:

objective =
  nDCG@20
  - 0.10 * top20_churn
  - 0.25 * top10_churn
  - 0.50 * top5_churn

Where:

topK_churn = fraction of rows whose topK set changed vs current 0.18348 ensemble

This makes the optimizer prefer changes that improve ranking without destroying the known anchor.

Use raw nDCG@20 as the final reported metric, but use anchored objective for hyperparameter search.

If A is positive, add small rank-band calibration.

score(q, t) =
  Σ_m [
    w_m_head * I(rank_m ≤ 5)   / (K + rank_m)
  + w_m_mid  * I(6 ≤ rank_m ≤ 20) / (K + rank_m)

w_m_mid  = w_m * (1 + δ_m_mid)
w_m_tail = w_m * (1 + δ_m_tail)

δ ∈ [-0.15, +0.15]

7. Why this should transfer better than rejected attempts
Better than category segment selection

Category segment selection failed because it made hard, high-variance decisions:

category X → use model A
category Y → use model B

The proposed method uses:

global weights first
small shrinked deltas second
nested validation always

So it cannot wildly swap models based on noisy category labels.

Better than train-session mixing

Direct train-session mixing likely hurt because it added distribution-specific memorization/noise. Anchored weighted RRF uses only the four honest LTR predictions already known to work OOF.

Better than seed-CF rank-20 rescue

Seed-CF changed 397/8000 rows but had flat dev and negative blind-like mean. That says the signal is too weak or too distribution-sensitive. Weighted RRF does not introduce a new weak source; it only corrects how much each already-useful LTR model contributes.

Better than full stacking

Full stacking may beat OOF, but it has many more parameters and can learn fold quirks. Anchored weighted RRF has maybe 6–20 effective parameters and direct metric tuning, so it is much harder to overfit.

1. 推荐路线

优先级如下：

尝试 LightGBM LambdaRank

最可能吃到非线性收益。

能自动利用：source rank interaction、intent gate、specificity、exact-match、seed similarity、popularity prior。

训练和推理代码更短。

但不要直接全量替换 BM25 排名。

同时保留 numpy/scipy pairwise ranker

作为无依赖 fallback。

适合做“保守排序校准器”。

线性模型可解释，容易加权重边界，比如强制 BM25/rule/exact-match 权重非负。

不要先做 pointwise logistic

pointwise logistic 可以当 baseline，但它优化的是“这个候选是不是 gold”，不是“同一 turn 内谁排前面”。

你这个数据是每个 group 一个或少数正例，pairwise 更自然。

2. 特征编码：重点是 rank 变换和 group 内标准化

每条 JSONL 应该至少有：

per-source rank 特征

不要直接只喂 rank=37。每个 source 做这组：

source_present = 0/1
source_recip = 1 / (rank + 60)      # RRF style
source_inv_rank = 1 / rank
source_log_rank = -log1p(rank)
source_top1 = rank <= 1
source_top3 = rank <= 3
source_top5 = rank <= 5
source_top10 = rank <= 10
source_top20 = rank <= 20

missing rank 不要填 999 后直接训练。更稳的是：

source_present = 0
source_recip = 0
source_inv_rank = 0
source_log_rank = 0
top flags = 0

这样模型不会把“missing=999”学成奇怪的线性数值。

legacy anchor 特征

BM25 是已知强 anchor，所以专门强化：

legacy_rank

legacy_top1/top3/top5/top10/top15/top20

这些能让模型知道“一个 item 是单源偶然命中，还是多源一致命中”。

对这些做 group 内 z-score 或 percentile：

因为不同 turn 的 BM25/RRF 分布不一样，原始分数跨 group 不可比。推荐同时保留：

可以用，但必须小心：

风险点：popularity 很容易把模型推回热门老歌，提升局部 dev 但伤 catalog diversity，也可能伤 broad intent。建议对 popularity 加上限，或者只允许在 tail rerank 中作为 tie-breaker。

3. 训练 / dev split

必须按 user/session 分组切分，不要按 row 随机切：

train sessions: 80%
dev sessions: 20%

原则：

同一个 session 的所有 turns 必须在同一 split
同一个 group_id 的所有 candidates 必须在同一 split
candidate generation 必须 train/dev 独立复现
dev 上不能 positive-inject 后再报主指标

如果你有 official train/dev：

train official train
tune official dev
blind-like validation = 从 train 内再切一个 session-level holdout

原因是你现在已经发现 local dev 和 blind 有明显 mismatch，所以不能只相信一个 dev 数。

4. 泄漏风险清单

最危险的是这些：

positive injection 泄漏
如果训练或 dev candidate pool 里强行注入 gold，可以用于诊断“reranker 上限”，但不能作为 honest end-to-end 指标。

track document augmentation 泄漏
只能用 train turns 给 track 文档加用户 query / assistant explanation / selected reason。不能用 dev gold，更不能用 blind 相关信息。

goal_progress 对齐错误
goal_progress_assessment[turn_number] 必须先确认它评价的是同 turn 的 music item，还是 transition 到下一个 turn。没确认前，只能把它作为弱反馈，不要强行当 positive/negative label。

future-history leakage
previously_recommended, user_feedback, session_seed 只能来自当前 turn 之前，不能看未来 turns。

track_id memorization
不要 one-hot track_id。可以用 smoothed train popularity，但不要让模型记住“这个 track 经常是答案”。

dev-tuned postprocess 过拟合
如果你对 dev 调了很多 alpha/K/gate，要再用一个 blind-like holdout 复核。

6. scipy/numpy pairwise ranker 方案

实现一个线性 RankNet 风格模型：

对每个 group，取 gold candidate p 和若干 negatives n，构造：

d = x_pos - x_neg
loss = log(1 + exp(-w · d)) + λ ||w||²

也就是希望：

w · x_pos > w · x_neg

负例采样优先取 hard negatives：

legacy top20 negatives
RRF top20 negatives
exact/source-consensus false positives
currently outranking gold negatives
random tail negatives

每个 positive 配 20–50 个 negatives 就够了。

推荐加入权重边界：

legacy_recip weight >= 0
rrf_score weight >= 0
rule_boost weight >= 0
exact_match weight >= 0
negative_feedback_match weight <= 0
previously_recommended weight <= 0
popularity weight in [-0.2, 0.2]

因为 scipy 的 L-BFGS-B 支持 bound-constrained optimization，所以这个很适合做“安全线性模型”。
SciPy 文档

推理时不要直接用：

final_score = model_score

而是用：

final_score = z_legacy + alpha * clip(z_model, -2, 2) + beta * exact_match_bonus

alpha 从小到大试：

0.05, 0.10, 0.15, 0.20, 0.30

只要 dev 上 top5/top10 开始掉，就不要再加。

7. 安全上线：legacy-head protection

我建议第一版提交级策略是：

K = 15
final[1:15] = legacy_bm25[1:15] unchanged
final[16:20] = model rerank from candidate pool excluding protected items

如果 dev 非常稳，再试：

K = 10
final[1:10] = legacy unchanged
final[11:20] = model rerank

不要一开始 rerank 全 top20。

允许进 head 的例外

只有这些情况允许模型把 item 提到 1–10：

title_exact_match and intent_exact_title
artist_exact_match and intent_artist
album_exact_match and intent_album
source_consensus_top20_count >= 3 and model_margin very high
negative_feedback demands replacing repeated/bad item

否则模型只能动 tail。

final ranking 公式

推荐：

base = legacy_recip_z
model = model_margin_z
exact = exact_entity_bonus
repeat = repeat_penalty
pop = clipped_popularity_prior

final_score =
    base
  + alpha * clip(model, -2, 2)
  + beta * exact
  - gamma * repeat
  + delta * pop

其中：

alpha = 0.05 ~ 0.20
beta = 0.2 ~ 0.8
gamma = 0.1 ~ 0.5
delta = 0.00 ~ 0.05

popularity 只做 tie-breaker，不要让它主导。

Experiment 2：linear pairwise ranker

评估：

如果 linear 模型全量 rerank 都不如 BM25，但 protect top15 后 nDCG 不掉，说明它仍可用于 tail improvement。

Experiment 3：LightGBM LambdaRank

装得上就跑：

small model: max_depth=3/4, num_leaves=7/15

Experiment 5：submission candidate

优先提交顺序：

最终建议

装 LightGBM，但不要赌 LightGBM。
真正低风险方案是：

LightGBM LambdaRank = main reranker
scipy pairwise linear ranker = dependency-free fallback
legacy BM25 top10/top15 = protected anchor
LTR model = tail reranker / exact-match promoter / tie-breaker

在你当前状态下，最可能安全涨分的不是“强模型全量重排”，而是：

BM25 top15 不动
16–20 用 LTR 从 multi-source candidates 中挑更像 gold 的
specific entity query 开 exact-match promotion
broad query 禁止模型大幅改 head
popularity 只做弱 prior
所有收益用 blind-like holdout 复核

0. 实验结论标准先定死

这次 shadow experiment 只回答：

在 BM25 top15 完全冻结的情况下，
embedding/CF 能否替换 16–20 位，
并且不损害 nDCG@20，同时提高 tail recall / catalog diversity？

上线候选只允许这类结果：

final[1:15] = legacy BM25[1:15] unchanged
final[16:20] = embedding/CF gated rerank

全量 rerank、top10 promotion、embedding 接管 BM25 head 都不做。

1. 使用的 retrieval channels

不要一次性把 7 个 embedding 全混进去。最小安全版本只用 4 个主通道：

Channel A：seed_metadata

用历史正向 seed tracks 的 metadata-qwen3 向量求均值。

query_vec = mean(L2(track_metadata_vec[positive_seed_tracks]))
score = dot(query_vec, track_metadata_vec[candidate])
topN = 200

作用：补 BM25 在自然语言语义上的同类歌曲召回。

Channel B：seed_attributes

用历史正向 seed tracks 的 attributes-qwen3 向量求均值。

query_vec = mean(L2(track_attributes_vec[positive_seed_tracks]))
score = dot(query_vec, track_attributes_vec[candidate])
topN = 200

作用：补 mood / genre / vibe / activity 类请求。

Channel C：seed_track_cf

用历史正向 seed tracks 的 track cf-bpr 向量求均值。

query_vec = mean(L2(track_cf_vec[positive_seed_tracks]))
score = dot(query_vec, track_cf_vec[candidate])
topN = 200

作用：协同过滤式“喜欢这个，也可能喜欢那个”。

Channel D：user_cf

如果当前 session / user 有 user cf-bpr：

score = dot(user_cf_vec[user], track_cf_vec[candidate])
topN = 200

作用：个性化 prior。

2. 第一版不要启用的 channels

这些先只做 shadow diagnostics，不进最终 tail rerank：

lyrics-qwen3
audio CLAP
image SigLIP2

原因不是它们没用，而是几小时内最容易引入噪声。
第一轮只在日志里看它们有没有 incremental recall。

可以记录：

lyrics_top200_contains_gold
audio_top200_contains_gold
image_top200_contains_gold

但不要让它们改提交结果。

后续只有在分组诊断明确有效时再启用：

lyrics-qwen3: 用户请求歌词、主题、叙事、words、meaning 时启用
audio CLAP: 用户请求 sound / energy / acoustic / instrumental / danceability 时启用
image SigLIP2: 默认不进 ranking，除非比赛数据明确显示封面视觉有强信号
3. seed track 提取规则

只用当前 turn 之前的信息。

positive_seed_tracks =
    previous music tracks in same session
    excluding tracks with explicit negative feedback
    excluding tracks after DOES_NOT_MOVE_TOWARD_GOAL if alignment is verified

如果 goal_progress_assessment 对齐还没完全确认，就不要强依赖它。更安全的第一版：

positive_seed_tracks = previous recommended / played tracks before current turn
negative_seed_tracks = only explicit user rejection, skip, dislike, not that, too X

最多取最近 5 个 seed：

recent weights:
last seed      1.00
2nd previous   0.80
3rd previous   0.60
4th previous   0.40
5th previous   0.30

如果没有 seed：

use user_cf only if available
else embedding system abstains

这个 abstain 很重要。不要为了覆盖率硬上 embedding。

4. candidate union

对每个 group / session-turn：

legacy_candidates = BM25 top100
metadata_candidates = seed_metadata top200
attributes_candidates = seed_attributes top200
track_cf_candidates = seed_track_cf top200
user_cf_candidates = user_cf top200

合并去重：

pool = union(
    legacy top100,
    metadata top200,
    attributes top200,
    track_cf top200,
    user_cf top200
)

但最终只能从 pool 里挑 16–20。

5. channel score normalization

每个 turn 内，每个 channel 单独归一化：

z_channel_score =
    (score - mean(topN_scores)) / std(topN_scores)

missing channel 记为 0，并加 present flag：

metadata_present
attributes_present
track_cf_present
user_cf_present

不要直接混 raw dot product，因为不同 embedding 通道的分布不可比。

6. 最小融合公式

第一版不用训练模型，直接 deterministic scoring：

embedding_score =
    0.35 * z_metadata
  + 0.25 * z_attributes
  + 0.25 * z_track_cf
  + 0.15 * z_user_cf

如果没有 seed，只剩 user_cf：

embedding_score = z_user_cf

最终 tail score：

tail_score =
    0.60 * legacy_tail_score_z
  + 0.30 * embedding_score
  + 0.10 * rule_boost_z
  - 0.20 * repeat_penalty

其中：

legacy_tail_score_z = BM25/RRF group-normalized score
repeat_penalty = same track / same artist overused in batch or session

这不是为了最强，而是为了安全。BM25 仍然主导。

7. freeze positions

做三个版本，但提交候选只看后两个：

Variant 0：shadow only
final = legacy BM25 top20 unchanged

只记录 embedding 是否能找到 gold。

用途：判断 embedding 有没有理论价值。

Variant 1：protect top15
final[1:15] = legacy[1:15]
final[16:20] = top 5 by tail_score from pool excluding final[1:15]

这是主实验。

Variant 2：protect top18
final[1:18] = legacy[1:18]
final[19:20] = top 2 by tail_score from pool excluding final[1:18]

这是极保守提交候选。

不要做：

protect top10
full rerank top20
embedding item enters top5

除非这个实验已经显著正收益。

8. hard gates：什么候选允许进 16–20

embedding 候选必须满足至少一个条件：

appears in legacy top100
or appears in at least 2 embedding channels top200
or user_cf rank <= 50
or seed_cf rank <= 50
or metadata/attributes rank <= 30

否则不准进最终 top20。

再加 negative filters：

if track was already recommended in same session: block
if artist explicitly rejected: block
if exact negative feedback match: block
if candidate has no metadata/id alignment check: block

这个能防止单通道偶然高分污染尾部。

9.1 incremental oracle recall

也就是：

如果这个几乎为 0，embedding 没有 tail value。

9.2 tail swap gain/loss

如果 lost_hits 明显大于 gained_hits，直接 reject。

9.3 protected nDCG delta
delta_ndcg20 = ndcg20_variant - ndcg20_legacy

这个是最终判定。

1. 当前策略判断

你已经观察到：

直接把 sampled train sessions 混入 LightGBM LTR training → held-out dev 变差

这说明至少有一个问题存在：

train/dev query distribution 不一致
train sampled sessions label pattern 太强，模型记忆了 train style
positive injection / hard-negative 构造方式污染了训练目标
track_id / popularity / train-context 特征被模型过度利用

所以现在应该改成：

train labels 不直接扩充 LTR 主训练集
train labels 只产生 auxiliary sources/features
这些 features 必须能单独 ablate、能 shadow、能关闭
最终排序仍保护 legacy BM25 head
2. Safe usage 1：train-context document augmentation

这是最值得保留的路径。

2.1 构建 track-context documents

对每个 train gold occurrence：

conversation_id
turn_id
gold_track_id
user_profile
conversation_goal
current user request
previous user feedback before current turn
previous positive seed tracks/artists
assistant/music history before current turn

生成该 gold track 的补充文档片段：

track_id -> [
  "user asked for ...",
  "goal mentions ...",
  "profile preference ...",
  "liked similar artist/genre/mood ...",
  ...
]

但是要拆成两个字段：

ctx_light: cleaned semantic context
ctx_raw: raw user/query snippets

推荐：

base_doc = title + artist + album + tags + metadata
aug_doc_light = base_doc + cleaned intent/mood/genre/activity/profile terms
aug_doc_raw = base_doc + capped raw train user snippets

BM25 source 分开建：

bm25_base
bm25_trainctx_light
bm25_trainctx_raw

不要直接把 augmentation 混进唯一 BM25 index。否则一旦 train-context 过拟合，你不知道是谁害的。

2.2 文本清洗规则

保留：

mood words
genre words
activity context
era / decade
language / region
energy / tempo adjectives
profile preference
explicit comparison like "similar to X"
positive feedback before current turn

降权或过滤：

exact title repeated too often
assistant boilerplate
template phrases
rare generated phrases
track-id-like artifacts
future-turn text
current gold response after recommendation

特别注意：如果 train assistant explanation 是围绕 gold track 生成的，它很容易把 “gold-specific wording” 写进文档。可以保留，但只放进 ctx_raw，权重低，不能进 head promotion。

2.3 每个 track 限制容量

不要让热门 track 的 train snippets 无限膨胀：

max_snippets_per_track = 20
max_tokens_per_track_context = 300~600
deduplicate near-identical snippets
prefer high-IDF / diverse snippets

否则 BM25 会更偏热门 track。

2.4 使用方式

推荐只把 train-context source 作为：

candidate generator
LTR feature
tail rerank source

不要让它直接覆盖 BM25 head。

安全融合：

final[1:15] = legacy BM25 unchanged
final[16:20] can use trainctx candidates

或者更保守：

final[1:18] = legacy BM25 unchanged
final[19:20] can use trainctx candidates
3. Safe usage 2：train-derived priors

train labels 可以变成弱统计 prior，但不能让它变成 track memorization。

3.1 可用 priors
track_popularity = count_train_gold(track)
artist_popularity = count_train_gold(artist)
album_popularity = count_train_gold(album)

intent_track_prior = count(track, intent)
intent_artist_prior = count(artist, intent)

category_track_prior = count(track, category)
specificity_track_prior = count(track, specificity)

profile_artist_prior = count(artist, profile_cluster)
turn_position_prior = count(track, turn_bucket)

但 track-level conditional prior 要非常弱，artist-level 通常更稳。

3.2 平滑

不要用 raw count。用：

log_track_pop = log1p(track_count)
log_artist_pop = log1p(artist_count)

smoothed_p(track | intent) =
    (count(track, intent) + alpha * p(track)) / (count(intent) + alpha)

建议：

alpha = 20~100

越怕过拟合，alpha 越大。

3.3 推理使用

prior 只能做 tie-breaker：

final_score =
    legacy_score_z
  + 0.05 ~ 0.10 * prior_z
  + other features

不要超过：

prior_weight > 0.15

否则它会把系统推向 train 热门歌，catalog diversity 和 blind 泛化都会受伤。

3.4 prior 的安全 gate

只在这些情况启用：

broad / vague / mood / genre query
legacy tail candidates 分数接近
candidate 已经被 BM25 或 embedding source 召回

不要在 exact title / exact artist query 中让 popularity prior 抢排序。

4. Safe usage 3：case-based distillation

这比直接混 train rows 更安全。

思路：

当前 dev/blind query -> 检索相似 train turns -> 聚合这些 train turns 的 gold tracks/artists/genres -> 作为一个 weak teacher source
4.1 建 train-turn index

每个 train turn 建一个 query document：

user_profile
conversation_goal
dialogue history before turn
current user request
positive/negative feedback before turn
intent/category/specificity

不要包含：

gold track title
gold artist
assistant answer after gold
future turns

每条 train-turn 存：

gold_track_id
gold_artist_id
gold_album_id
gold_genre/tags if available
4.2 对 dev/blind turn 检索相似 train turns
similar_train_turns = BM25_or_embedding_search(current_turn_text, topK=20~50)

聚合：

case_track_score[t] = sum(sim_i for train_i if gold_track_i == t)
case_artist_score[a] = sum(sim_i for train_i if gold_artist_i == a)
case_album_score[al] = sum(sim_i for train_i if gold_album_i == al)

然后给 candidate 加 features：

candidate_case_track_score
candidate_case_artist_score
candidate_case_album_score
candidate_case_max_sim
candidate_case_support_count
4.3 更推荐 artist/tag distillation，不推荐 exact track distillation

exact track distillation 容易记忆 train label。

更稳的使用顺序：

artist score > album score > tag/genre score > exact track score

如果 exact track 只在一个相似 train turn 中出现，不要强推。

安全条件：

case_support_count >= 2
or max_sim very high and candidate already in BM25 top100
4.4 使用位置

case-based teacher 只允许影响：

positions 16–20
or LTR feature
or response explanation hint

不允许直接把 train gold track 拉进 top5。

5. Safe usage 4：sample weighting，而不是 sample mixing

如果你还想让 LTR 用 train sessions，先不要“随机采样混入”。改成 group-level sample weighting。

5.1 只用 naturally retrieved candidates

训练 LTR 时区分：

natural candidate: retrieval pipeline 本来能召回
positive-injected candidate: 为了训练强行加入 gold

推荐：

natural positives weight = 1.0
positive-injected-only positives weight = 0.1 or 0

如果 gold 不在自然 candidate pool，LTR 学它没有意义；那是 retrieval recall 问题，不是 ranking 问题。

5.2 dev-like weighting

对每个 train group 计算输入侧特征：

turn_number
query length
intent
category
specificity
number of entities
has seed tracks
has profile
legacy top20 score distribution
BM25 top1/top5 score gap

训练一个简单 classifier 区分：

train-like vs dev-like

注意：这个 classifier 只能用 input features，不能用 label/gold。

得到 density weight：

w = p(dev_like | x) / p(train_like | x)
w = clip(w, 0.25, 2.0)

如果不想这么复杂，直接用 stratified caps：

每个 intent/category/turn_bucket 的训练样本数量按 dev 分布重采样
5.3 noisy train examples 降权

降权：

gold only appears because of positive injection
query extremely generic
assistant/history text contains template artifacts
train context has near-duplicate generated wording
track is ultra-popular and appears in too many train labels

升权：

exact entity queries
clear positive feedback turns
has seed and gold is semantically close to seed
hard negative clearly outranks gold in legacy

但是不要过度升权热门 track。

6. Dev reporting leakage guards

这是最重要的部分。

6.1 两套评估必须分开

你需要明确输出两类结果：

HONEST DEV:
  augmentation/prior/distillation built from train only
  dev gold never used
  no positive injection in reported end-to-end metric

ORACLE/DIAGNOSTIC:
  may include positive injection or dev-gold analysis
  only用于定位问题
  not comparable to leaderboard expectation

报告里不要混。

6.2 train OOF augmentation

如果你在 train 内做 ablation，不能用全 train augmentation 评估 train fold。必须 K-fold：

split train conversations into K folds by session/user

for fold k:
    build train-context docs from train folds except k
    evaluate on fold k

否则你会把某个 fold 的 gold context 写进它自己的 track doc，train OOF 分数虚高。

6.3 official dev / blind 构建规则

对 official dev：

build all artifacts from official train only
evaluate on official dev

对 blind：

build all artifacts from official train + allowed non-label metadata only
run on blind input

如果你决定把 official dev labels 也折入 final train artifact，必须单独标记：

not valid for dev reporting
only for final blind submission candidate

但在 blind mismatch 很大时，我不建议太早把 dev labels 折进去。

6.4 时间泄漏

对每个 turn t，只能用：

profile
goal
conversation history before or at current user request
previous music tracks
previous user feedback

不能用：

future user turns
future assistant responses
current gold response
goal_progress label if it actually refers to future transition

如果 goal_progress_assessment[turn_number] 对齐还不确定，只能当弱特征，不要当强 positive/negative label。

6.5 文件 manifest

每次生成 artifact 写 manifest：

JSON
{
  "artifact": "bm25_trainctx_light",
  "source_split": "official_train_only",
  "excluded_splits": ["dev", "blind"],

}

没有 manifest 的结果不进入报告。

8. 判断 train data 是否真的有帮助
8.1 必须看 incremental recall

对每个 train-derived source：

legacy_miss = gold not in legacy top20
source_hit = gold in source topK
incremental_recall = count(legacy_miss and source_hit) / count(all groups)

如果：

trainctx_light top100 incremental recall ≈ 0
case_teacher top100 incremental recall ≈ 0

说明它没提供新候选，只是在扰动排序。

8.2 必须看 tail swap precision

对 protect top15 / top18：

gained_hits = legacy top20 missed, new top20 hits
lost_hits = legacy top20 hit, new top20 misses
net_hits = gained_hits - lost_hits

接受条件：

net_hits >= 0
lost_hits <= gained_hits
nDCG@20 >= legacy - 0.001
catalog diversity >= legacy

如果只涨 diversity 不涨 net hits，那是 perturbation，不是 ranking improvement。

8.3 分组诊断

至少按这些切：

exact title / artist / album query
genre / mood query
similar-to-seed query
negative feedback query
vague query

turn 1–2
turn 3–5
turn 6–8

has profile
no profile

has previous seed
no previous seed

gold popularity high / mid / low

典型判断：

trainctx 对 mood/genre 有收益，exact query 掉分
=> trainctx 只能在 non-exact intent 启用

case teacher 只在 has-seed late turns 有收益
=> only enable when seed_count >= 1 and turn >= 3

prior 提升 vague query，但伤 specific query
=> prior only for broad/vague intent
8.4 过拟合信号

直接 rollback 的信号：

train OOF 提升，但 official dev 掉
full-train augmentation > fold-out augmentation 很多
source 主要提升 ultra-popular tracks
exact entity query 被 train prior 改坏
new hits 集中在少数 train-frequent tracks
candidate 被 trainctx 拉进 top20，但 BM25_base / embedding / entity 都不支持
randomized train labels 也能带来类似提升

最后一条很重要：做一个 sanity check。

把 train gold track 随机打乱后重建 trainctx/prior
如果效果还差不多，说明你学到的是 popularity 或 artifact，不是真正语义。

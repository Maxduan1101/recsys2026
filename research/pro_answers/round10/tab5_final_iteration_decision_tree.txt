Remaining optimization directions worth doing now
1. Response-only A/B between judge_clean_mix and lexplus

This is the highest-value remaining test because the ranking is identical. You can isolate the response effect cleanly.

Run exactly these two variants:

Variant	Ranking	Response	Why test
A	weighted four-model RRF	judge_clean_mix	current primary, less template-heavy
B	same as A	lexplus	higher dev lexical and higher Blind-A Distinct-2

Decision basis should not be “Distinct-2 only.” Use Distinct-2 as a positive signal, but penalize visible compact-template repetition. A response system can look diverse by n-gram count while still sounding formulaic across sessions.

Worth doing now: make a minimal lexplus-polish variant only if it is cheap. Do not rebuild the response system. The useful edit is to keep lexplus ranking/output content but add a small anti-template rule:

avoid repeating the same opening phrase across many turns;

vary connector phrases;

keep recommendations grounded in the user’s stated mood/artist/genre;

avoid overly short “template capsule” responses when the dialogue context is rich;

preserve exact track IDs and ranking.

Call this variant:

lexplus_softened

Only submit it if it changes responses without changing ranking or JSON validity.

2. Final ranking stability and tie-break audit

Do not add new retrieval sources. Instead, audit the existing weighted four-model RRF stack.

Check:

deterministic sorting after RRF;

stable tie-break order;

no accidental dependence on dictionary order, multiprocessing order, or cache order;

no duplicate track IDs within a turn;

no invalid track IDs;

all required turns present;

all inference uses the full all_tracks catalog, not a filtered catalog. The official baseline README explicitly warns that filtering the catalog during inference may invalidate evaluation.

This is worth doing because final leaderboard losses often come from mechanical drift, not model quality.

3. Very small RRF weight perturbation only around the current winner

A final micro-sweep is worth doing only if it is bounded.

Allowed:

current weights;

±5% and ±10% relative perturbations around each of the four model weights;

one leave-weakest-model-downweighted variant;

one “sharpen top source” variant if one model dominates top-1 dev wins.

Not allowed:

adding a new model;

re-indexing the corpus;

adding unvalidated embeddings;

changing candidate generation depth unless you already know the failure mode.

The goal is not discovery. The goal is to confirm the current weighted RRF is not sitting on a brittle local accident.

4. Position-sensitive top-k audit

Because recommendation metrics usually reward top positions most strongly, inspect cases where your candidate appears in positions 2–20 but not position 1.

Worth doing now:

test a tiny top-rank override only when one source has a very confident exact title/artist match.

Do not change the whole ranking. Only consider a guarded rule like:

If exact artist/title/album evidence exists in the user turn or state, and the candidate appears in at least one model’s top 1–3, allow it to move above generic consensus tracks.

This is only worth doing if your dev errors show exact-match misses.

Directions to stop
Stop 1. Consensus fallback for Blind A

Stop working on it for the current public-feedback loop.

Reason: it helps dev but changes 0/80 Blind-A cases. That means it cannot improve Blind-A public feedback unless Blind-A preprocessing or gating changes. Keep the code only if it might affect Blind B later, but do not count it as a current optimization.

Decision:

Disable it as a claimed final improvement unless it changes Blind-B rows or a local blind simulation.

Stop 2. Chasing dev lexical diversity alone

The lexplus gain is real numerically:

dev lexical: +0.00502 absolute

Blind-A Distinct-2: +0.02588 absolute

But lexical diversity is not the same as response quality. Compact-template-heavy output can be penalized if the judge rewards conversational usefulness, grounding, or naturalness.

Decision:

Use lexical/Distinct-2 as a tie-breaker, not the sole selector.

Stop 3. Large new rerankers, LTR, embeddings, or learned models

At final-iteration stage, stop adding infrastructure that is not already validated end-to-end.

Stop:

new LTR training;

new embedding retrieval;

new neural reranker;

new LLM judge reranking;

new feature families;

large prompt rewrites;

new augmentation sources.

Reason: these can improve dev while creating hidden Blind-A/B distribution risk, invalid IDs, latency/caching failures, or irreproducibility.

Stop 4. Broad corpus/index changes

Do not change the BM25 corpus construction now unless you found a deterministic bug.

Stop:

adding/removing metadata fields globally;

changing tokenization;

changing normalization;

changing stopword handling;

changing train-context augmentation rules;

changing candidate pool size broadly.

Reason: corpus changes can shift many rankings at once and are hard to diagnose with limited public feedback.

Stop 5. Optimizing components that do not affect the submitted file

Do not optimize logs, diagnostics, consensus fallback, feature export, or embedding scaffolding unless they change final predictions or final responses.

A simple rule:

If it does not change the final JSON, it is not a final-iteration optimization.

Stop 6. Blind-A overfitting by repeated tiny submissions

Public feedback is useful, but it is also a trap. If you repeatedly tune to Blind A, you risk selecting a variant that wins Blind A but loses Blind B or final hidden scoring.

Use public feedback only to choose between pre-declared candidates, not to invent new ones after seeing every result.

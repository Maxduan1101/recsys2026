2. Template-bank design

Use a compositional template bank, not one large list of whole responses. This gives controlled diversity while preserving grounding.

Banks
Bank	Purpose	Example variants
opener	Acknowledge the request or feedback	“For that {mood} direction,” / “Sticking with your {genre} thread,” / “To move away from the last pick,”
history_bridge	Connect to prior likes/dislikes	“since you responded well to {liked_tag} tracks” / “after the {negative_tag} pick felt off”
track_intro	Present recommendation	“I’d start with {title} by {artist}” / “Try {title} — {artist}”
metadata_reason	Ground explanation	“it is tagged {tag_phrase}” / “its {year} release keeps it in the {era} window” / “the album context, {album}, matches your {album_constraint} cue”
contrast_or_next	Explain refinement	“This keeps the energy lower than {prev_track}.” / “It gives you a cleaner entry point before the deeper cuts.”
closer	Optional short close	“That makes it a safer first play.” / “I’d put this near the top of the queue.”

A single response is assembled from 3–5 clauses. For example:

{opener} {track_intro}. {metadata_reason}; {history_bridge}. {closer}
Template classes by conversation state

Use different templates for different user intents:

Specific lookup: user is trying to identify one song/album/artist.
Response should be cautious: “I’d try X first” unless there is an exact match.

Mood/activity request: user wants music for studying, gym, rain, relaxation.
Ground in tags such as relaxing, ambient, energetic, dance, acoustic, downtempo.

Era/genre request: user asks for 90s rock, early-2000s punk, classic jazz.
Ground in release_date, tags, artist, album.

More like previous: user likes a prior item.
Mention shared tags, same artist, same album, same decade, or similar catalog cluster.

Negative feedback/refinement: user rejects a previous item.
Explicitly say how the new pick moves away from the rejected attribute.

Exploration/discovery: user asks for “new artists” or broad discovery.
Mention why the top pick opens a lane, then optionally name one contrast pick.

4. How to use profile, history, and feedback
Current user utterance: strongest signal

Extract explicit constraints first:

User: “I want something rainy, soft, reflective, with acoustic elements.”

Planner facts:

JSON
{
  "mood": ["rainy", "soft", "reflective"],
  "instrument_or_texture": ["acoustic"],
  "avoid": [],
  "specificity": "high"
}

Response should visibly reuse one or two of those words.

Session history: second strongest signal

Track previous recommended item IDs and map them back to metadata. If the user gives positive feedback, promote shared tags. If the user gives negative feedback, demote or avoid those tags.

Examples:

“that was too heavy” → avoid tags: heavy, metal, loud, aggressive
“I liked the piano but want less sleepy” → keep: piano; avoid: sleepy, slow, ambient
“more like the last one” → prefer shared artist/tag/album/decade
Profile: weak but useful signal

Use profile for ranking and evidence selection, but be careful about mentioning it. Do not say “because you are a 20-year-old female listener…” That feels creepy and may be judged poorly. Safer uses:

Good: “This stays close to your indie-folk leaning.”
Risky: “Because you are a Canadian woman in your 20s…”

Mention profile only when it is musically meaningful, such as preferred_musical_culture, and only if that field is explicitly available in the input.

Conversation goal: use if exposed

If conversation_goal.listener_goal is available, treat it as a high-level intent summary, but still prioritize the latest user turn. The public dataset viewer shows conversation goals like “discover multiple mellow and pleasant songs” or “find one specific Nine Inch Nails song.”

5. Metadata grounding rules

Every response should contain:

At least one track title and artist from predicted_track_ids.

At least one user-specific reason from current utterance, history, or profile.

At least one metadata-backed fact from title, artist, album, tag, year, duration, or popularity.

No unsupported audio/lyric/biography claims.

Good:

“For the mellow, easy-listening mood, I’d start with ‘With Rainy Eyes’ by Emancipator. It is tagged relaxing, piano, ambient, and downtempo, so it fits the calm lane without needing a big stylistic jump.”

Bad:

“You’ll love the emotional lyrics and warm analog production.”

Unless lyrics or production are in the metadata or user text, that is hallucinated.

Use “tagged as” for tag-derived claims:

“it is tagged as ambient and downtempo”

rather than:

“it has a spacious ambient mix”

The first is safely metadata-grounded; the second infers audio details.

6. Self-check rules

Run these checks after candidate generation.

Entity whitelist

Reject any named entity not found in:

current user utterance
previous dialogue turns
recommended track metadata
previously recommended track metadata
user profile fields

This prevents invented artist names, albums, genres, and external facts.

Claim-source ledger

For each non-generic adjective, require a source:

JSON
{
  "relaxing": "track.tag_list",
  "downtempo": "track.tag_list",
  "early 2000s": "release_date or user_request",
  "Black Keys": "user_request or artist_name"
}

If no source exists, delete or soften the claim.

Contradiction checks

Reject if:

user says “not too heavy” and response says “keeps the heavy sound”
user asks for 2000s and response says a 1991 track “matches the 2000s era”
user disliked last artist and response says “since you liked that artist”
Sensitive profile check

Do not mention age, gender, or country unless the user explicitly brought it up. These fields may help modeling, but they rarely improve response quality.

Minimum quality check

Require:

20–55 words
title + artist present
one explicit user-intent phrase
one metadata-backed reason
no duplicate track IDs
no unsupported claims

The official submission format requires predicted_track_ids and predicted_response, with track IDs valid and ordered by relevance.

9. Practical inference algorithm
Python
运行
def generate_response(example, predicted_track_ids, catalog, global_bigram_counts):
    state = parse_dialogue_state(example.conversations)
    profile = parse_profile(example.user_profile)

    top_tracks = [catalog[tid] for tid in predicted_track_ids[:3]]
    evidence = build_evidence_ledger(state, profile, top_tracks)

    candidates = []
    for template in select_template_classes(state, evidence):
        for variant in template.variants:
            text = fill_template(variant, evidence)
            if passes_self_checks(text, evidence, state):
                candidates.append(text)

    if not candidates:
        candidates = [fallback_response(evidence)]

    return max(
        candidates,
        key=lambda r: (
            personalization_score(r, state, profile),
            metadata_grounding_score(r, evidence),
            feedback_alignment_score(r, state),
            novel_bigram_ratio(r, global_bigram_counts),
            -hallucination_risk(r, evidence),
            -length_penalty(r),
        )
    )

The fallback should still be grounded:

“I’d start with “{title}” by {artist}. Its catalog metadata gives the closest match I found to your latest request.”

Even this is better than a universal “Here are some songs you might enjoy.”

What I would build first

Build one fused LightGBM ranker where candidate generation is intentionally broad, and ranking is where precision happens.

The official evaluator requires one prediction per session-turn, up to 20 unique track IDs, ordered by relevance. 
GitHub
 The baseline instructions also warn that inference must retrieve candidates from the entire track catalog using all_tracks, not a filtered split. 
GitHub
 So the architecture should be:

all-catalog candidate generation → feature table → LightGBM ranker → conservative diversity tail rerank → response generation.

1. Candidate pool per session-turn

For each (session_id, turn_number), build a union of candidates, then deduplicate:

BM25 sources you already have

Use all current BM25 multi-index outputs:

latest user utterance → track name, artist, album, tags

full dialogue context → track metadata index

train-context augmented retrieval

shifted feedback seed retrieval

source-specific top-K lists

Keep the top 100–300 per source, not only the final BM25 top-20.

Embedding candidates

Add lightweight FAISS or numpy top-K retrieval:

Text-query → track metadata / attributes / lyrics embeddings using a query embedding from the same Qwen3 embedding family if available.

Seed-track → track-track nearest neighbors using metadata, attributes, lyrics, audio, and CF-BPR vectors.

For later turns, compute max/mean similarity from candidate to prior accepted or “moves toward goal” seeds.

For first turns, use conversation goal + latest user utterance + profile musical culture as the query text.

CF/user candidates

Add:

top-K by dot(user_cf_bpr, track_cf_bpr)

top-K within same high-level artist/tag neighborhood as the user’s known history

top-K from user’s frequent artists/tags, but only as recall candidates, not hard filters

Keep candidate pool size around 500–2000 per turn. If this is too large, cap per source but preserve source diversity.

2. Training labels and groups

Train one ranking group per session-turn.

For each group:

positive: ground-truth track for that turn

negatives: union candidates excluding the positive

hard negatives: top BM25 misses, dense nearest misses, same-artist/same-tag wrong tracks, popular tracks

random negatives: small sample from catalog to teach calibration

During training, inject the positive if missing so the ranker can learn. During validation, separately report:

candidate recall@20/100/500/1000

oracle nDCG@20 from the candidate pool

actual nDCG@1/10/20 after ranking

This separation matters: if candidate recall is weak, LightGBM cannot recover.

Use LightGBM LGBMRanker with objective="lambdarank" or objective="rank_xendcg", metric="ndcg", and eval_at=[1,10,20]. LightGBM’s LambdaRank docs require integer labels where higher means more relevant, and mention label gain / truncation controls. 
LightGBM

3. Feature set

Use features in five blocks.

A. Retrieval-source features

For each source:

raw score

normalized score within query

rank

reciprocal rank

top-1/top-5/top-20 source flags

number of sources retrieving candidate

max / sum / mean normalized source score

source family flags: title, artist, album, tags, train-context, seed-shift, dense, CF

These are likely your highest-value features because your “source diagnostics” already encode which retrieval path works by turn type.

B. Exact and fuzzy music-entity features

exact title mention in latest user message

exact artist mention

exact album mention

normalized edit similarity for title / artist / album

quoted span match

year / decade match

genre and mood tag overlap

explicit negative constraint hit, e.g. “not too heavy,” “no live version”

candidate was already recommended in session

same artist/album as already rejected track

same artist/album as positively received track

C. Conversation and feedback features

turn number

goal specificity/category if available

number of previous tracks

previous track feedback: MOVES_TOWARD_GOAL, DOES_NOT_MOVE_TOWARD_GOAL

candidate similarity to last positive seed

candidate similarity to last negative seed

candidate’s artist/tag overlap with positive seeds minus negative seeds

last user utterance length and specificity signals: contains artist, title, era, mood, activity, lyrics fragment

The dataset includes multi-turn conversations, user profiles, conversation goals, and goal-progress assessments, so these fields are available for training-time supervision and feature engineering.

Validate by session/user-grouped split, not random rows. Track performance by turn number and goal type. Your current public blind nDCG@20 of 0.1935 is already well above the official BM25 dev baseline nDCG@20 of 0.0815, though public blind and dev are not directly comparable.

 That means the easy BM25 wins are probably mostly harvested; the ranker should target reordering and cross-source fusion.

Diversity rerank without sacrificing nDCG

Do not globally randomize top-20. Use a conservative two-zone policy:

positions 1–10: pure LightGBM score, except remove duplicates and previously shown tracks

positions 11–20: apply mild diversity penalty

Example final score for tail positions:

final_score =
    lgbm_score
  - 0.03 * same_artist_count_in_list
  - 0.02 * same_album_count_in_list
  - 0.02 * log1p(global_track_usage_so_far)
  + 0.01 * source_diversity_bonus

Also add a global “tail-only usage cap”:

never cap a track if it is ranked top-10 by LGBM

for tail slots, prefer high-scoring tracks not already used many times globally

avoid repeated same-artist clusters unless the query explicitly asks for that artist

This should improve catalog diversity with minimal nDCG risk, because catalog diversity is global unique coverage across all predictions.

Response work plan

Because blind text is judged separately from recommendation accuracy on personalization and explanation quality, treat response generation as a scored module, not filler. 
推荐系统挑战赛

Use a metadata-grounded response generator:

Extract user constraints from the latest utterance and conversation history: artist, genre, mood, era, activity, lyrics clue, “more like X,” “not X.”

Take top 1–3 ranked tracks and their metadata: title, artist, album, tags, release date.

Generate 45–90 words.

Mention at least one concrete track and artist.

Explain why the list matches the user’s current request or prior feedback.

Avoid hallucinated facts not present in metadata.

Rotate templates and phrasing to improve Distinct-2.

Example template pattern:

Given your request for {mood/genre/era} and the earlier signal around {seed_track_or_artist},
I’d start with {track_1} by {artist_1}. It matches through {tag_or_album_or_year_reason}.
I also included {track_2} and {track_3} to keep the set close to that sound while giving you a bit of range.

Make 30–60 variants of this structure with different openings, transitions, and closings. Lexical diversity is Distinct-2, so repeated generic responses will score poorly. 
GitHub

Best practical approach:

Use deterministic templates for reliability.

Optionally use a local LLM only to paraphrase within strict metadata constraints.

Run post-checks: response non-empty, mentions real recommended track/artist, no fake claims, length under a safe cap, no repeated boilerplate.

Expected effect: nDCG unchanged, lexical diversity likely large improvement if current responses are repetitive, and hidden judge likely improves if responses become specific and personalized.

Where embedding retrieval and CF fit

I would not build “embedding retrieval” as a separate final submission before the ranker. I would build it as candidate expansion + features.

Minimum viable embedding layer:

Build FAISS indexes for:

metadata Qwen3 embeddings

attributes Qwen3 embeddings

lyrics Qwen3 embeddings

CF-BPR track vectors

optionally audio CLAP for seed-track neighbors

For each turn:

encode query text once

retrieve top 200 from metadata/attributes/lyrics

retrieve top 200 from user CF dot product

retrieve top 100 neighbors from positive feedback seeds

Add all candidates to the LightGBM feature table.

Minimum viable CF/user layer:

feature: dot(user_cf_bpr, track_cf_bpr)

feature: rank by user CF

feature: user-history artist overlap

feature: user-history tag overlap

feature: candidate popularity within user country/age group if enough data

fallback: set missing user-vector features to 0 and include missing flags

This is much safer than using CF as a standalone ranker because conversational intent can override long-term taste.

Why not cross-encoder next

A cross-encoder can help when the remaining problem is fine semantic matching among already-good candidates. Your current state suggests the bigger remaining wins are:

fusing many existing signals;

adding dense/CF recall;

fixing scored response/diversity axes.

A cross-encoder reranking 50–100 candidates per turn may be worth testing later as a single feature:

cross_encoder_score(latest_user + compact_history, track_title + artist + album + tags)

Then feed that score into LightGBM. I would not let it directly replace the ranker unless it proves clear dev nDCG@20 gains and does not break latency.

Decision rule from diagnostics

Before coding everything, compute these three numbers on dev:

Diagnostic	Meaning	Action
recall@1000 of current BM25 union	Can the target be found at all?	If low, add embedding/CF candidates before ranker.
oracle_nDCG@20 from candidate pool	Best possible nDCG if ranker were perfect.	If high, LightGBM is the obvious next step.
gap: oracle_nDCG@20 - actual_nDCG@20	Ranking headroom.	Big gap means reranking has high ROI.

Rule of thumb:

If recall@1000 < 0.45, first add embedding/CF retrieval.

If recall@1000 ≥ 0.45 and actual nDCG is far below oracle, train LightGBM immediately.

If text/diversity are near-zero, patch response/diversity before the next blind submission even if the ranker is not finished.

Final priority order

LightGBM rank_xendcg / LambdaRank fusion reranker over current BM25 + dense + CF candidates.

Response and diversity patch in the same submission cycle.

Embedding retrieval as candidate expansion and ranker features.

CF/user features as candidate expansion and ranker features.

Cross-encoder only after the above, preferably as a top-50 feature rather than the main ranker.

The fastest strong submission is therefore: BM25/embedding/CF candidate union → LightGBM ranker → top-10 relevance-preserving + tail diversity rerank → metadata-grounded personalized responses.

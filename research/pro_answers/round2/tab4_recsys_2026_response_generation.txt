Below is a ranking-safe response-generation subsystem whose goal is to raise Distinct-2 and blind Gemini judge scores while keeping predicted_track_ids byte-for-byte unchanged.

The official task asks systems to output both a top-20 ranked list and a natural-language response; response quality is judged separately from recommendation accuracy, with blind Gemini judging Personalization and Explanation Quality while the exact judge prompt is hidden. The public evaluator computes catalog diversity from recommended track IDs and lexical diversity as Distinct-2, so a post-ranking generator can improve text metrics without touching nDCG@20 or catalog diversity.

 The evaluator’s Distinct-2 implementation lowercases each response, whitespace-tokenizes it, counts unique bigrams over all responses, and divides by total bigrams, which makes controlled anti-boilerplate generation especially important.

1. Core design: freeze ranking, generate from an evidence pack

Use a two-lane architecture:

ranker/reranker
  -> frozen_top20_track_ids
  -> EvidencePack builder
  -> response planner
  -> template/LLM candidate generator
  -> factual verifier
  -> lexical-diversity selector
  -> final predicted_response

2. Template families

Use a family router rather than one universal template. Each family has 8–20 surface variants, but all variants share the same factual slots.

Family	Use when	Response structure	Example skeleton
Goal-match explanation	User states mood/genre/style clearly	Acknowledge request → name 1–3 anchors → explain shared tags	“For the {requested_mood/style} you asked for, I’d start with {track1} by {artist1}. Its catalog tags point toward {tag1}/{tag2}, and {track2} keeps the {shared_tag} thread going.”
Specific-search / identification	User is trying to find a specific song, artist, album, era, or remembered clue	State strongest candidate → mention uncertainty if needed → give backups	“The closest match is {track1} by {artist1}: it lines up with your {artist/era/album/mood} clue. I also included {track2} and {track3} as nearby fallbacks.”
Feedback repair	User rejected or corrected earlier picks	Contrast previous issue → say how list shifts → cite evidence	“Since the last direction seemed {too_soft/too_broad}, I shifted this set toward {new_constraint}. {track1} is the sharpest match, while {track2} keeps a little of the earlier {old_tag} flavor.”
Feedback continuation	User liked prior recommendation or asked for more	Maintain continuity → add controlled novelty	“I’m continuing from the part you responded to: {liked_anchor}. These picks stay close to {shared_tag}, with {track3} adding a slightly different {tag} angle.”
Discovery / exploration	Broad listener goal, low specificity, “new artists,” “anything like…”	Use one familiar anchor plus two exploratory tracks	“I used {known_artist_or_tag} as the compass, then widened the set with {track1} and {track2}. They share {shared_tag}, but the artists give you a broader discovery path.”
Era / album / scene-aware	Request mentions years, decade, album, subculture	Cite release_date or album only when known	“I leaned into the {era/album} clue here. {track1} fits the date/album evidence, and {track2} keeps the same {genre_tag} lane without drifting too far.”
Mood balancing	User asks for complex affect: “melancholic but hopeful,” “calm but not sleepy”	Explain trade-off between tags	“I balanced {mood_a} with {mood_b}: {track1} covers the {tag_a} side, while {track2} and {track3} keep the set from getting too {undesired_quality}.”
Uncertainty-aware recommendation	Metadata is weak or user request is ambiguous	Be honest, still recommend	“The metadata does not pin this down perfectly, so I’m giving you the closest matches rather than pretending certainty. {track1} best matches {evidence}; {track2} is a safer adjacent option.”
List-guided concise	Later turns, user just wants tracks	Short but personalized	“I’d go with {track1}, {track2}, and {track3}. They stay aligned with your {goal_phrase} request through {shared_tags}, while the full list gives you a few adjacent directions.”
Question-plus-recommendation	User goal is underspecified but ranking still required	Recommend now, ask one useful follow-up	“I picked these around {best_inferred_goal}. {track1} is the clearest fit from the tags; tell me whether you want the next round more {axis_a} or more {axis_b}.”

The important part is that “variety” comes from discourse function, not random adjectives. A response can be lexically diverse and judge-friendly because it references different goal states, feedback signals, artists, tags, and uncertainty patterns.

3. Controlled variation that improves Distinct-2 without looking fake

Distinct-2 rewards unique bigrams globally, but the blind judge likely penalizes weird, bloated, or unhelpful text. Treat Distinct-2 as a constraint, not the sole objective. LLM-as-a-judge systems are known to have reliability issues and biases such as verbosity and position effects, so do not assume “longer and more ornate” is better.

Variation knobs

Use these knobs in a controlled way:

Openers:
“Given your last cue…”, “I leaned into…”, “For this round…”, “To keep the search aligned…”, “This set is built around…”, “I’m treating your request as…”

Evidence verbs:
“points toward”, “lines up with”, “leans on”, “keeps close to”, “widens from”, “anchors the list in”, “balances”, “backs up”

Recommendation framing:
“start with”, “lead with”, “make {track} the anchor”, “use {track} as the safest match”, “put {track} first because…”

Closers:
“then the rest of the list explores nearby territory”, “with a few backups in case you want a broader pass”, “so you get both a clear match and some discovery room”, “without drifting away from the mood you asked for”

Length bands:
Rotate 35–50, 50–70, and 70–85 words by turn/family. Very short responses can boost Distinct-2 mechanically, but the Gemini judge is supposed to evaluate personalization and explanation quality, so most final responses should contain at least one clear “because” relation. 
推荐系统挑战赛

Global n-gram governor:
Keep a running set of bigrams already used in the prediction file. Penalize candidates that repeat high-frequency boilerplate bigrams such as “you might,” “I recommend,” “these tracks,” “based on,” “your preferences,” “great fit.” Since the evaluator uses lowercase whitespace tokens, this can be reproduced exactly offline. 
GitHub

Do not game punctuation:
Because tokenization is whitespace-based, artificial punctuation or random rare words could inflate Distinct-2. Avoid this. It may improve the public lexical number while hurting hidden judge quality.

4. Personalization features to feed the planner

Create “personalization atoms” with provenance:

JSON
[
  {"text": "asked for early-2000s alternative rock", "source": "latest_user_utterance"},
  {"text": "preferred musical culture: North American Indie Folk / Americana", "source": "user_profile"},
  {"text": "liked the previous mellow acoustic direction", "source": "feedback_classifier"},
  {"text": "goal: discover multiple songs for quiet reflection", "source": "conversation_goal"},
  {"text": "top recommendation tags: acoustic, folk, mellow", "source": "top20_metadata"}
]

The response planner should require at least two personalization atoms when available:

Python
运行
personalization_coverage =
    1[mentions_latest_request] +
    1[mentions_goal_or_specificity] +
    1[uses_feedback_signal] +
    1[uses_profile_musical_culture] +
    1[explains top track using metadata]

Practical rules:

Use latest user utterance first. It is more judge-visible than long-term profile fields.

Use conversation goal to choose response style. For HH or highly specific goals, use identification/uncertainty-aware templates. For LL broad discovery goals, use exploration templates. For HL, explain multiple constraints. For LH, explain the broad clue but keep uncertainty.

Use feedback only when explicit. Say “since you asked for something less heavy” only if the user actually did. Do not say “since you liked…” unless there is positive feedback or a goal-progress signal that is available at inference time.

Use profile sparingly. “This stays near your indie-folk listening profile” is useful. “As a 20-year-old listener from Brazil…” is usually awkward and may look stereotyped.

Use listening-history summaries, not raw demographics. Derive top artists/tags from the user’s history, but only mention them when they overlap with the current request or top-20 evidence.

5. Hallucination controls

The response should be metadata-grounded but not metadata-dumpy. Use a claim ledger.

Claim typing

Every phrase in the response should map to one of these:

Claim type	Allowed?	Example
Direct metadata	Yes	“{track} by {artist}”
Derived from metadata	Yes, if deterministic	“early-2000s” from release_date=2003-...
Tag-based description	Yes, with soft wording	“catalog tags lean mellow/acoustic”
User-stated preference	Yes	“you asked for something dramatic”
History/profile summary	Yes, if computed	“your profile leans toward indie folk”
External music knowledge	No	“this was a Grammy-winning hit”
Audio property not in metadata	No	“fast drums,” “distorted guitars,” “female vocal” unless present in tags/conversation
Lyrics/meaning	No	“the lyrics are about…”
Popularity interpretation	Avoid	“popular” only if the metadata field is intentionally used and calibrated

Prefer hedged metadata language: “the tags point toward,” “the catalog marks it as,” “this lines up with the metadata,” rather than absolute claims like “this track is definitely dreamy.”

Generator prompt
You write one short response for a music recommender.

Use ONLY the evidence JSON. Do not use outside music knowledge.
Do not mention lyrics, chart history, instruments, tempo, vocalist gender,
cover art, or artist biography unless those exact facts appear in evidence.

Mention 1–3 tracks from the frozen top-20 list.
Explain why the recommendations fit the user's latest request, goal, or feedback.
Use natural language; avoid generic boilerplate.
Do not change, add, remove, or reorder track IDs.
Return only the response text.
Verifier prompt
You are a factual verifier for a music recommendation response.

Inputs:
1. Evidence JSON
2. Candidate response

Return JSON:
{
  "supported": true/false,
  "unsupported_claims": [],
  "mentioned_tracks_not_in_top20": [],
  "mentioned_artists_not_in_top20": [],
  "used_personalization_atoms": [],
  "metadata_claims": [
    {"claim": "...", "source_field": "..."}
  ],
  "revision": "..."
}

Rules:
- A claim is supported only if it follows from the evidence JSON.
- Tags support soft descriptions only, not precise audio analysis.
- Release dates support year/decade claims.
- Do not allow facts from general music knowledge.

Run the verifier either as deterministic code plus an LLM, or as two independent LLM checks with fallback to a conservative template when they disagree.

9. Practical rollout plan

Start with a template-only version. It should already improve Distinct-2 by rotating families, openers, verbs, and closers while remaining safe.

Then add a local rewriter with a narrow instruction: “rewrite this grounded draft without adding facts.” The LLM should not receive the full catalog or external context, only the EvidencePack and draft.

Then add the verifier. This is the highest-leverage safety component because it lets you use more varied generation without letting hallucinations leak into the final JSON.

Finally add candidate selection using Distinct-2 delta plus personalization/explanation heuristics. Use Gemini only as a shadow judge or light reranker, not as the sole generator, because the official blind prompt is undisclosed and overfitting to a guessed rubric can backfire.

10. Recommended default policy

For the challenge setting, I would use:

1. Freeze top-20 recommendations.
2. Generate 6 candidates:
   - 4 template-family candidates
   - 2 local-LLM rewrites of the best template drafts
3. Verify all candidates.
4. Select candidate maximizing:
   personalization + explanation + distinct2_delta - unsupported_claims - boilerplate.
5. Fallback to conservative template if no candidate passes.

This gives you a clean story for the paper: ranking is held constant, response generation is a grounded post-processing layer, lexical diversity is improved through controlled discourse variation, and judge quality is improved through explicit use of user goal, conversation feedback, and metadata-supported explanations.

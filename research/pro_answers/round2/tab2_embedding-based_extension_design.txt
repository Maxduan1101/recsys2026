Recommendation: add a BM25 + calibrated seed/user embedding hybrid reranker

The first extension I would build is not a new dense query encoder and not a learned ranker yet. Use your existing BM25 output as the lexical anchor, then add four cheap embedding signals:

seed_metadata, seed_attributes, seed_cf, and user_cf.

This is minimal because it only needs the official precomputed embeddings, your TrackEmbeddingStore, a seed linker, exact dot products over ~47k tracks, and a hand-tuned linear fusion. It is strong because it improves both recall through candidate union and ranking through complementary content/CF signals, while keeping BM25 dominant when the dialogue contains exact artist/track/album constraints.

This fits the challenge setup: Music-CRS is explicitly multi-turn, uses TalkPlayData-Challenge, and provides pre-extracted multimodal track and user embeddings.

Core design

For each conversation turn:

Run the existing BM25 retriever over all_tracks.

Extract high-confidence seed tracks from dialogue history and available context.

Build three seed-based dense query vectors:

seed_metadata: centroid/max similarity in metadata-qwen3_embedding_0.6b.

seed_attributes: centroid/max similarity in attributes-qwen3_embedding_0.6b.

seed_cf: item-item similarity in cf-bpr.

Add one user-based dense signal:

user_cf: raw BPR user-track dot product when the user embedding is available.

Take a candidate union from BM25 and each embedding source.

Score every union candidate with robustly normalized source scores.

Final rank with a simple gated linear fusion.

Return top 20 unique valid track IDs.

The track embedding dataset exposes all_tracks with about 47.1k rows and the relevant channels: audio-laion_clap, image-siglip2, cf-bpr, attributes-qwen3_embedding_0.6b, lyrics-qwen3_embedding_0.6b, and metadata-qwen3_embedding_0.6b; the first extension should use only metadata, attributes, and CF to keep risk low. 
Hugging Face
 The user embedding dataset has cf-bpr user vectors with train/test-warm/test-cold splits, so user_cf must be optional and mask-aware. 
Hugging Face

Why these four signals first
1. seed_metadata

This is the safest embedding add-on after BM25.

Use it when the dialogue contains seed tracks, artists, albums, or previous positively received recommendations that can be mapped to track IDs. It should capture semantic similarity around track/artist/album metadata, while BM25 still handles exact lexical matches.

Use cosine similarity over L2-normalized metadata embeddings:

q_seed_metadata = normalize(weighted_mean(metadata_norm[seed_track_ids]))
score_seed_metadata(track) = dot(metadata_norm[track], q_seed_metadata)

For multiple seeds, do not rely only on the centroid. Add a max-similarity feature:

centroid_score = dot(track_vec, centroid)
max_seed_score = max_j dot(track_vec, seed_vec_j)

seed_metadata_raw = 0.7 * centroid_score + 0.3 * max_seed_score

The max term helps when a session contains multiple tastes, such as “I like Radiohead and also want something like SZA.”

2. seed_attributes

This should be the second seed signal. It is likely more useful for mood, genre, instrumentation, vibe, and taste descriptors than pure metadata.

Use the same centroid plus max approach:

q_seed_attributes = normalize(weighted_mean(attributes_norm[seed_track_ids]))
seed_attributes_raw = 0.7 * centroid_cos + 0.3 * max_seed_cos

Expected win: turns where BM25 retrieves exact words but misses semantically similar tracks.

Expected risk: attributes may overgeneralize and pull in “vibe-near” but lexically irrelevant songs. Keep its weight below or equal to seed_metadata initially.

3. seed_cf

Use cf-bpr track vectors for item-item collaborative similarity from seed tracks.

For seed-to-track similarity, prefer cosine over L2-normalized CF vectors at first:

q_seed_cf = normalize(weighted_mean(track_cf_norm[seed_track_ids]))
seed_cf_raw = dot(track_cf_norm[track], q_seed_cf)

Why cosine here: item-item BPR dot products can be strongly affected by vector norms and popularity. Cosine is a safer minimal item-neighborhood signal. Later, ablate raw item-item dot product separately.

Use seed_cf only when you have at least one strong seed; ideally give it full weight only with two or more seeds:

g_seed_cf = min(1.0, effective_seed_count / 3.0)
4. user_cf

Use raw BPR user-track dot product:

user_cf_raw(track) = dot(user_cf_raw[user_id], track_cf_raw[track])

This is different from seed_cf. For user-track relevance, raw BPR dot product is the natural first choice because BPR-style matrix factorization is usually trained to score user-item pairs by dot product. Robust normalization will handle scale differences.

Use this only when the user vector exists. For missing/cold users, set user_cf_available = 0 and remove its weight entirely. Do not impute with zeros and then treat the score as meaningful.

Seed extraction policy

Keep seed extraction high precision. A bad seed is worse than no seed because it drifts all dense channels.

Use seed sources in this priority order:

Explicit track IDs already present in system state, history, or previous recommendations.

Exact normalized (track_name, artist_name) matches from the dialogue.

Exact track title matches only when the title is unambiguous.

Previous recommendations that the user accepted or reacted to positively.

Optional later: high-confidence artist-level matches expanded to that artist’s known tracks.

Do not use BM25 top results as pseudo-seeds in the first version. That creates feedback loops: BM25 makes a mistake, embeddings amplify it.

Seed weights:

explicit_liked_track        1.00
accepted_previous_reco      0.80
mentioned_track_exact       0.70
profile/history seed        0.50
ambiguous title-only match   skip initially
negative/rejected mention    skip or use later as negative seed

Apply recency decay:

weight *= 0.85 ** turns_ago

Cap to the most recent/highest-confidence 5–10 seeds.

Candidate union

Use candidate union, not reranking BM25-only candidates. Otherwise embeddings can only reorder BM25’s mistakes and cannot fix recall.

Starting values:

BM25 topK              500
seed_metadata topK     300
seed_attributes topK   300
seed_cf topK           300
user_cf topK           500
max union size         usually 800–1,800 after de-dup
final output           top 20

Robust score normalization

Do not directly add BM25, cosine, and BPR scores. Their scales are incompatible.

Use per-turn, per-source robust normalization. For each source, calibrate using the source’s own retrieved topK scores, not all union candidates only.

Recommended function:

Python
运行
def robust01(raw_scores, calibration_scores, eps=1e-8):
    """
    raw_scores: scores for union candidates for one source
    calibration_scores: scores from that source's own topK retrieval
    returns: [0, 1] normalized scores
    """
    cal = calibration_scores[np.isfinite(calibration_scores)]
    cal = cal[cal > -np.inf]

    if len(cal) < 5:
        return np.zeros_like(raw_scores, dtype=np.float32)

    lo = np.percentile(cal, 5)
    hi = np.percentile(cal, 95)

    if hi - lo < eps:
        # fallback: rank-only source is safer than unstable score scaling
        return np.zeros_like(raw_scores, dtype=np.float32)

    y = (raw_scores - lo) / (hi - lo + eps)
    y = np.clip(y, 0.0, 1.0)
    y[~np.isfinite(raw_scores)] = 0.0
    return y.astype(np.float32)

Then blend robust score with a rank feature:

Python
运行
def rank01(rank, topk):
    if rank is None:
        return 0.0
    return 1.0 - min(rank - 1, topk - 1) / max(topk - 1, 1)

Final per-source normalized feature:

source_feature = 0.75 * robust_score_norm + 0.25 * rank01

This prevents one source from dominating because of raw-score scale while still rewarding tracks that were genuinely high-ranked by that retriever.

For BM25, use log1p(bm25_raw) before normalization:

bm25_raw_for_norm = log1p(bm25_raw)

For dense cosine sources, use raw cosine. For user_cf, use raw BPR dot product, then robust-normalize.

Initial fusion formula

Start with a transparent hand-tuned fusion:

final_score =
    1.00 * bm25
  + 0.35 * g_seed      * seed_metadata
  + 0.30 * g_seed      * seed_attributes
  + 0.20 * g_seed_cf   * seed_cf
  + 0.25 * g_user_cf   * user_cf
  + 0.05 * source_agreement

Where:

g_seed    = min(1.0, effective_seed_count / 2.0)
g_seed_cf = min(1.0, effective_seed_count / 3.0)
g_user_cf = 1.0 if user_cf exists else 0.0

source_agreement = number_of_sources_with_norm_score_above_0.5 / number_of_available_sources

Add one protective gate:

if bm25 top score is very strong and query contains exact track/artist constraints:
    reduce user_cf weight by 50%

Rationale: user_cf reflects general taste, but a turn like “play songs by Daft Punk” should not be overwhelmed by the user’s historical preference vector.

def topk_cosine(self, channel: str, query_vec: np.ndarray, k: int):
    X = self.norm[channel]          # shape [n_tracks, dim], float32 or float16
    mask = self.mask[channel]       # shape [n_tracks]

    q = query_vec.astype(np.float32)
    q = q / (np.linalg.norm(q) + 1e-8)

    scores = X @ q
    scores = scores.astype(np.float32)
    scores[~mask] = -np.inf

    idx = np.argpartition(-scores, kth=min(k, len(scores)-1))[:k]
    idx = idx[np.argsort(-scores[idx])]
    return idx, scores[idx], scores

def topk_user_cf(self, user_vec_raw: np.ndarray, k: int):

Step 2: implement seed query construction
Python
运行
def build_seed_query(store, channel, seed_track_indices, seed_weights):
    valid = store.mask[channel][seed_track_indices]
    idx = np.asarray(seed_track_indices)[valid]
    w = np.asarray(seed_weights, dtype=np.float32)[valid]

    if len(idx) == 0:
        return None

    X = store.norm[channel][idx].astype(np.float32)
    w = w / (w.sum() + 1e-8)

    q = (X * w[:, None]).sum(axis=0)
    q = q / (np.linalg.norm(q) + 1e-8)
    return q

For metadata and attributes, use the corresponding Qwen3 embedding channels. For seed_cf, use normalized cf-bpr.

Memory and performance

At this catalog size, exact dense search is fine. You do not need FAISS first.

Approximate memory for the full track embeddings:

metadata 1024 dims
attributes 1024 dims
lyrics 1024 dims
audio 512 dims
image 768 dims
cf 128 dims
total ≈ 4480 dims / track
tracks ≈ 47k

All six channels as one float32 matrix would be roughly:

47,071 * 4,480 * 4 bytes ≈ 804 MB

If you store both raw and normalized float32 versions, that is roughly 1.6 GB before overhead. For the first extension, you only need metadata, attributes, and CF:

47,071 * (1024 + 1024 + 128) * 4 bytes ≈ 391 MB

Raw plus normalized float32 for these three channels is around 780 MB. With float16 storage, about half that.

Practical recommendation:

Store normalized matrices as C-contiguous float32 memmaps if CPU BLAS speed matters.

Store raw CF as float32.

Store unused channels — lyrics/audio/image — unloaded for now.

Use argpartition for topK, then sort the selected topK.

Batch dense scoring by channel when running many turns:

Python
运行
scores = Q @ X.T

where Q is [batch_size, dim].

For ~47k tracks, metadata or attributes exact scoring is only about 48 million multiply-adds per query. That is acceptable on a modern CPU and trivial on GPU. The complexity cost is much lower than adding an LLM reranker.

Main failure modes and guards
Bad seed linking

Problem: wrong track matched from ambiguous title.

Guard: high-precision seed linker only. Require artist match when possible. Skip ambiguous title-only matches.

Query drift from previous recommendations

Problem: your own bad recommendations become future seeds.

Guard: only use previous recommendations as seeds if the user reacted positively or the dialogue state marks them as accepted. Otherwise give them very low weight or skip.

user_cf overwhelms the current request

Problem: the user asks for a specific genre/artist, but user history pulls general favorites.

Guard: cap user_cf weight and reduce it when BM25 has a strong exact-match signal.

CF popularity/norm bias

Problem: raw BPR scores favor high-norm popular tracks.

Guard: use raw dot only for user-track user_cf; use cosine for item-item seed_cf. Robust-normalize both. Ablate raw item-item dot later.

Multi-intent conversations

Problem: centroid averaging collapses “I like A and B” into a bland middle.

Guard: include max_seed_score alongside centroid score. Consider per-seed max more important as seed diversity increases.

Missing embeddings

Problem: channels have empty lists or invalid masks.

Guard: every source score must be mask-aware. Missing channel means source_available = 0, score = 0, and no weight contribution.

Candidate union hurts precision

Problem: embeddings add many plausible but irrelevant candidates.

Guard: keep BM25 weight dominant. Add dense candidates, but require robust normalization and source agreement. Inspect top false positives by source.

Turn leakage or invalid catalog usage

Problem: accidentally using labels, future turns, or non-all_tracks subsets.

Guard: seed extraction must use only inference-time-visible context. Retrieval must be over the full all_tracks catalog, as the baseline README warns.

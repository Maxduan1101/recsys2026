1. 推荐路线

优先级如下：

尝试 LightGBM LambdaRank

最可能吃到非线性收益。

能自动利用：source rank interaction、intent gate、specificity、exact-match、seed similarity、popularity prior。

训练和推理代码更短。

但不要直接全量替换 BM25 排名。

同时保留 numpy/scipy pairwise ranker

作为无依赖 fallback。

适合做“保守排序校准器”。

线性模型可解释，容易加权重边界，比如强制 BM25/rule/exact-match 权重非负。

不要先做 pointwise logistic

pointwise logistic 可以当 baseline，但它优化的是“这个候选是不是 gold”，不是“同一 turn 内谁排前面”。

你这个数据是每个 group 一个或少数正例，pairwise 更自然。

2. 特征编码：重点是 rank 变换和 group 内标准化

per-source rank 特征

不要直接只喂 rank=37。每个 source 做这组：

source_present = 0/1
source_recip = 1 / (rank + 60)      # RRF style
source_inv_rank = 1 / rank
source_log_rank = -log1p(rank)
source_top1 = rank <= 1
source_top3 = rank <= 3
source_top5 = rank <= 5
source_top10 = rank <= 10
source_top20 = rank <= 20

missing rank 不要填 999 后直接训练。更稳的是：

source_present = 0
source_recip = 0
source_inv_rank = 0
source_log_rank = 0
top flags = 0

这样模型不会把“missing=999”学成奇怪的线性数值。

legacy anchor 特征

BM25 是已知强 anchor，所以专门强化：

这些能让模型知道“一个 item 是单源偶然命中，还是多源一致命中”。

3. 训练 / dev split

必须按 user/session 分组切分，不要按 row 随机切：

train sessions: 80%
dev sessions: 20%

原则：

同一个 session 的所有 turns 必须在同一 split
同一个 group_id 的所有 candidates 必须在同一 split
candidate generation 必须 train/dev 独立复现
dev 上不能 positive-inject 后再报主指标

如果你有 official train/dev：

train official train
tune official dev
blind-like validation = 从 train 内再切一个 session-level holdout

原因是你现在已经发现 local dev 和 blind 有明显 mismatch，所以不能只相信一个 dev 数。

4. 泄漏风险清单

最危险的是这些：

positive injection 泄漏
如果训练或 dev candidate pool 里强行注入 gold，可以用于诊断“reranker 上限”，但不能作为 honest end-to-end 指标。

track document augmentation 泄漏
只能用 train turns 给 track 文档加用户 query / assistant explanation / selected reason。不能用 dev gold，更不能用 blind 相关信息。

goal_progress 对齐错误
goal_progress_assessment[turn_number] 必须先确认它评价的是同 turn 的 music item，还是 transition 到下一个 turn。没确认前，只能把它作为弱反馈，不要强行当 positive/negative label。

future-history leakage
previously_recommended, user_feedback, session_seed 只能来自当前 turn 之前，不能看未来 turns。

track_id memorization
不要 one-hot track_id。可以用 smoothed train popularity，但不要让模型记住“这个 track 经常是答案”。

dev-tuned postprocess 过拟合
如果你对 dev 调了很多 alpha/K/gate，要再用一个 blind-like holdout 复核。

6. scipy/numpy pairwise ranker 方案

实现一个线性 RankNet 风格模型：

对每个 group，取 gold candidate p 和若干 negatives n，构造：

d = x_pos - x_neg
loss = log(1 + exp(-w · d)) + λ ||w||²

也就是希望：

w · x_pos > w · x_neg

负例采样优先取 hard negatives：

legacy top20 negatives
RRF top20 negatives
exact/source-consensus false positives
currently outranking gold negatives
random tail negatives

每个 positive 配 20–50 个 negatives 就够了。

推荐加入权重边界：

legacy_recip weight >= 0
rrf_score weight >= 0
rule_boost weight >= 0
exact_match weight >= 0
negative_feedback_match weight <= 0
previously_recommended weight <= 0
popularity weight in [-0.2, 0.2]

因为 scipy 的 L-BFGS-B 支持 bound-constrained optimization，所以这个很适合做“安全线性模型”。
SciPy 文档

推理时不要直接用：

final_score = model_score

而是用：

final_score = z_legacy + alpha * clip(z_model, -2, 2) + beta * exact_match_bonus

alpha 从小到大试：

0.05, 0.10, 0.15, 0.20, 0.30

只要 dev 上 top5/top10 开始掉，就不要再加。

7. 安全上线：legacy-head protection

我建议第一版提交级策略是：

K = 15
final[1:15] = legacy_bm25[1:15] unchanged
final[16:20] = model rerank from candidate pool excluding protected items

如果 dev 非常稳，再试：

K = 10
final[1:10] = legacy unchanged
final[11:20] = model rerank

不要一开始 rerank 全 top20。

允许进 head 的例外

只有这些情况允许模型把 item 提到 1–10：

title_exact_match and intent_exact_title
artist_exact_match and intent_artist
album_exact_match and intent_album
source_consensus_top20_count >= 3 and model_margin very high
negative_feedback demands replacing repeated/bad item

否则模型只能动 tail。

final ranking 公式

推荐：

base = legacy_recip_z
model = model_margin_z
exact = exact_entity_bonus
repeat = repeat_penalty
pop = clipped_popularity_prior

final_score =
    base
  + alpha * clip(model, -2, 2)
  + beta * exact
  - gamma * repeat
  + delta * pop

其中：

alpha = 0.05 ~ 0.20
beta = 0.2 ~ 0.8
gamma = 0.1 ~ 0.5
delta = 0.00 ~ 0.05

popularity 只做 tie-breaker，不要让它主导。

Experiment 0：冻结基线

先固定三个东西：

candidate exporter

legacy BM25 submission reproduction

如果 gold 不在 candidate pool，reranker 没救，只能靠 retrieval。

Experiment 2：linear pairwise ranker

先实现 scipy pairwise ranker。

评估：

all-candidate rerank
protect top10
protect top15
protect top20 unchanged sanity

如果 linear 模型全量 rerank 都不如 BM25，但 protect top15 后 nDCG 不掉，说明它仍可用于 tail improvement。

Experiment 3：LightGBM LambdaRank

装得上就跑：

small model: max_depth=3/4, num_leaves=7/15
medium model: max_depth=5, num_leaves=31

只接受满足以下条件的模型：

protect top15 nDCG@20 >= legacy - tiny_tolerance
exact-intent subset improves
broad/mood subset不明显崩
catalog diversity不恶化
Experiment 4：gate by intent

分别测：

specific title/artist/album query
genre/mood/context query
similar-to-seed query
negative-feedback query
vague query

推荐规则：

specific intent: allow exact-match promotion
broad intent: protect top10/15, model only tail
negative feedback: allow demote repeated/disliked artist
vague intent: almost fully trust BM25/popular prior, tiny model alpha
Experiment 5：submission candidate

优先提交顺序：

1. BM25 head unchanged + better response generator
2. protect top15 + model rerank 16-20
3. protect top10 + model rerank 11-20 only if blind-like dev stable
4. full model rerank only as diagnostic, not first submission
最终建议

装 LightGBM，但不要赌 LightGBM。
真正低风险方案是：

LightGBM LambdaRank = main reranker
scipy pairwise linear ranker = dependency-free fallback
legacy BM25 top10/top15 = protected anchor
LTR model = tail reranker / exact-match promoter / tie-breaker

在你当前状态下，最可能安全涨分的不是“强模型全量重排”，而是：

BM25 top15 不动
16–20 用 LTR 从 multi-source candidates 中挑更像 gold 的
specific entity query 开 exact-match promotion
broad query 禁止模型大幅改 head
popularity 只做弱 prior
所有收益用 blind-like holdout 复核

1. 先定义一个固定 anchor

把你现在 public-safe 的 BM25/head20 ranking 当 anchor：

anchor = head20_compactresp_v2
ranking = public-safe BM25 top20 unchanged
known Blind A nDCG@20 = 0.1935
known catalog_diversity ≈ 0.0257 / 0.0258

以后所有实验都做 paired comparison：

delta_i = metric(candidate, row_i) - metric(anchor, row_i)

不要看 candidate 单独分数。你现在的问题是 dev 分布不等于 Blind A 分布，所以 absolute dev nDCG 没意义，paired delta on blind-like rows 才有意义。

2. 构造 blind-like local holdout
Step A：从 Blind A input 反推抽样分布

对 Blind A 80 rows 做同一套特征解析，不需要 label：

row features:
- turn_number: exact turn 1..8
- turn_bucket: early / mid / late
  early = 1-2
  mid   = 3-5
  late  = 6-8

- request_category:
  exact_track
  artist_or_album
  genre_style
  mood_activity
  era_language_region
  similar_to_seed
  feedback_refinement
  broad_exploration
  other

- specificity:
  S3 = exact entity request
       e.g. title, artist, album, quoted track, “by X”
  S2 = multi-constraint
       e.g. genre + mood, era + artist, activity + language
  S1 = broad but meaningful
       e.g. “something upbeat”, “jazz recommendations”
  S0 = weak / ambiguous / pure feedback
       e.g. “not this”, “something else”, “continue”

然后统计：

P_blind(turn_number)
P_blind(category)
P_blind(specificity)
P_blind(turn_bucket, category, specificity)

关键是不要只按 turn stratify。Blind A 只有 80 rows，某些 turn 可能多，但真正影响 ranking 的往往是 specificity：exact artist/title 查询和 broad mood 查询完全不是同一种检索问题。

Step B：从本地 8000-turn dev 里抽 80-row shadow panels

构造候选池：

eligible_pool = all labeled local rows
               excluding any rows used for training / augmentation / LTR fitting
               with valid gold track
               with same inference format as Blind A

如果 Blind A 每个 session 只抽一个 current turn，本地 panel 也强制：

max 1 row per session per panel

如果 Blind A 有 session 重复，则匹配它的 session multiplicity distribution。

然后采样 200 到 1000 个 shadow panels，每个 panel 80 rows：

For each panel seed:
    for each stratum c = (turn_bucket, category, specificity):
        quota_c = round_largest_remainder(80 * P_blind(c))
        sample quota_c rows from eligible_pool[c]
    if cell too small:
        backoff order:
          1. same turn_bucket + category
          2. same category + specificity
          3. same turn_bucket
          4. global blind-like pool

至少保留三类 panel：

A. exact-blind panels
   Match Blind A joint distribution as closely as possible.

B. smoothed-blind panels
   Laplace smooth rare strata:
   P = 0.8 * P_blind + 0.2 * P_local_selected_like

C. stress panels
   Force fragile groups:
   - exact entity
   - broad mood/activity
   - late feedback/refinement
   - low-specificity S0/S1

最终提交决策看：

primary = exact-blind panels
robustness = smoothed + stress panels

4. 决策指标：不要被 catalog diversity 误导

你给的 Blind A 设置是：

80 rows × top20 = 1600 recommendation slots
catalog size = 47071
max possible catalog_diversity = 1600 / 47071 ≈ 0.0340

你现在：

head20 unique tracks ≈ 1216
catalog ≈ 1216 / 47071 ≈ 0.0258

head15/head18 tail diversity 这种方案如果把 unique tracks 提到 1348：

catalog ≈ 1348 / 47071 ≈ 0.0286
absolute gain ≈ +0.0028

按你推测的 composite：

composite ≈ 0.5*nDCG
          + 0.1*catalog
          + 0.1*lexical
          + 0.3*((judge-1)/4)

那么 catalog +0.0028 只贡献：

0.1 * 0.0028 = +0.00028 composite

这意味着只要 nDCG 掉：

0.5 * ΔnDCG < -0.00028
=> ΔnDCG < -0.00056

catalog gain 就被抵消了。

所以：tail diversity 只能在几乎零 nDCG 损失时提交。 你之前看到 dev nDCG 从 0.08587 掉到 0.08180，哪怕这个全量 dev 不 blind-like，也已经说明它不是天然安全方案。

5. 三个候选的提交判定
Candidate 1：head20 response-only

这是当前最安全的一类。

ranking unchanged
=> ΔnDCG@20 = 0
=> Δcatalog = 0 or tiny
=> only bet response lexical / judge quality

提交条件：

must pass:
- ranking exactly identical to anchor on all Blind A rows
- no row has changed top20 order
- local Distinct-2 clearly higher
- bad/noisy tag leakage < 2%
- hallucinated artist/title/genre < 1%
- response mentions grounded profile/history/feedback when available
- duplicate response template rate < 5%

Decision:

Submit first unless response generator has obvious hallucination/tag-leak problems.

原因很简单：public anchor 的 nDCG 已经强，response-only 不会破坏它，而官方评测包含回复质量。

Candidate 2：head18 tail diversity

这里假设只改 positions 19-20，或者最多 18-20。它可以试，但必须过非常严格的 nDCG 安全线。

在 200+ shadow panels 上要求：

median ΔnDCG@20 >= -0.0005
mean ΔnDCG@20   >= -0.0005
P10 ΔnDCG@20    >= -0.0020
P25 ΔnDCG@20    >= -0.0010

median gold_lost_top20 <= 0 rows per 80-row panel
P90 gold_lost_top20    <= 1 row per 80-row panel

exact_entity ΔnDCG >= -0.001
S3 specificity ΔnDCG >= -0.001
artist_or_album ΔnDCG >= -0.001

Diversity side must be meaningfully positive:

unique tracks per 80-row panel +5% minimum
catalog_diversity +0.0015 absolute minimum
artist repeat count down at least 5-10%

But even then，提交顺序也应该是：

head20 response-only first
head18 tail diversity second only if nDCG risk is basically flat

如果 shadow panels 结果类似：

ΔnDCG = -0.002 or worse
catalog +0.002 to +0.003

不要提交。catalog 权重太小，补不回来。

Candidate 3：future LTR / embedding changes

不要直接和 head20 submission 比最终榜分。先分两阶段验证。

Phase 1：candidate recall only

对 embedding / CF / LTR 前的 candidate union 看：

gold_in_candidates@50
gold_in_candidates@100
gold_in_candidates@200
gold_in_candidates@500

在 blind-like panels 上要求：

gold_in_candidates@100 +3pp minimum
gold_in_candidates@200 +5pp preferred
no drop in BM25 candidate coverage

如果只是召回多了但 ranking 会乱，先不要进 top20。

Phase 2：protected reranking

LTR/embedding 一开始只能影响尾部：

mode A: rerank positions 16-20 only
mode B: rerank positions 11-20 only
mode C: rerank positions 6-20 only
mode D: full rerank, last resort

上线阈值：

For positions 16-20 rerank:
    median ΔnDCG >= 0
    P10 ΔnDCG >= -0.002
    exact_entity ΔnDCG >= -0.001
    gold_lost_top20 P90 <= 1

For positions 11-20 rerank:
    median ΔnDCG >= +0.0015
    P10 ΔnDCG >= -0.0015
    top10 gold demotion = 0 or near 0

For positions 6-20 rerank:
    median ΔnDCG >= +0.003
    lower 80% bootstrap CI > 0
    no major group negative

For touching top5:
    median ΔnDCG >= +0.005
    P10 ΔnDCG >= 0
    exact_entity and S3 both positive

LTR/embedding 特别要看这些 group：

likely helps:
- broad_exploration
- mood_activity
- similar_to_seed
- late feedback/refinement

likely hurts:
- exact_track
- artist_or_album
- high-specificity S3

所以最安全策略是：

if category in {exact_track, artist_or_album} or specificity == S3:
    preserve BM25 top10 or top20

if category in {mood_activity, broad_exploration, similar_to_seed} and specificity <= S2:
    allow embedding/LTR insertion in positions 16-20 first
6. Final experiment order

按这个顺序跑，不要乱：

E0 anchor
   current public-safe BM25 head20

E1 head20 response-only
   same ranking, improved response generator

E2 head19 tail diversity

E3 head18 tail diversity

   only positions 16-20, gated by category/specificity

E7 LTR protected rerank
   preserve top10 or top15, rerank tail

E8 LTR broader rerank
   only if shadow panels strongly positive

For each experiment produce:

all-turn dev metrics
blind-like exact panels
blind-like smoothed panels
stress panels
grouped diagnostics
row-level loss audit

The actual decision should be based on this priority:

1. Blind-like exact panel paired ΔnDCG
2. high-specificity / exact-entity group safety
3. gold_lost_top20 count
4. catalog / lexical gains
5. all-turn dev score

All-turn dev is last. It is useful for finding bugs, not for selecting submissions.

7. Practical submission recommendation

Given your current facts:

head20 response-only:
    ranking unchanged
    nDCG risk ≈ zero
    likely improves lexical / judge
    submit first

head18 or head15 tail diversity:
    only submit if blind-like panels show ΔnDCG basically flat
    catalog gain alone is too small to justify even tiny nDCG loss

future LTR / embeddings:
    validate as recall/features first
    deploy only behind top10/top15 protection
    start with positions 16-20

My default rule for your next submission choice:

Submit head20 response-only first.

Submit head18 tail-diversity only if:
    median ΔnDCG >= -0.0005
    P10 ΔnDCG >= -0.002
    exact/S3 groups are not negative
    gold_lost_top20 <= 1 row in at least 90% of panels
    catalog_diversity gain >= +0.0015

Otherwise keep the BM25 head and spend the next iteration on response quality + protected embedding/LTR tail features.

1. What Gemini-style judge likely rewards

The response should visibly answer:

1. Did the assistant understand this user’s current request?
2. Did it use the conversation/profile/history when available?
3. Did it explain why the recommended tracks fit?
4. Is the explanation grounded in track/artist/metadata, not hallucinated?
5. Does it sound like a music assistant, not a metric optimizer?

The challenge asks systems to use dialogue history and user context to produce both ranked track recommendations and response text, so the response should make that connection explicit. 
RecSys

Your response should usually contain:

- 1 direct acknowledgement of the current need
- 1 personalization hook
- 1-2 named tracks/artists from top-ranked recommendations
- 1 grounded explanation using safe metadata
- 1 light diversity / ordering explanation

Target length:

35-70 words
1 paragraph
2 sentences usually
3 sentences max

Too short looks generic. Too long risks hallucination and repeated filler.

2. Core response skeleton

Use this as the main template family:

{acknowledge_request}. I’d start with {track1} by {artist1} and {track2} by {artist2} because {grounded_reason_1}. Since you {profile_or_feedback_hook}, I also included {track3_or_style_bucket} to {explain_variety_or_progression}.

Example shape:

You asked for something warmer and less aggressive, so I kept the list melodic rather than high-energy. I’d start with “Track A” by Artist A and “Track B” by Artist B because they match the indie-pop / mellow tags in this turn. Since you reacted well to softer picks earlier, I also mixed in a few adjacent tracks that keep the same mood without repeating the same artist.

This is better than:

Here are some diverse recommendations that match your taste and provide a fresh listening experience.

because the first version contains current-turn grounding, track evidence, and conversation memory.

C. Feedback refinement

Use when the user says “not that”, “more upbeat”, “less mainstream”, “something else”.

I adjusted away from {negative_feedback} and leaned more toward {positive_direction}. {track1} by {artist1} should be the safest next try because {grounded_reason}. I kept {track2} and nearby picks in the list to explore the same direction without repeating the earlier mismatch.

This is judge-friendly because it proves the system used dialogue feedback.

D. Broad exploration
I treated this as a discovery request, so the list balances familiar anchors with a few adjacent finds. {track1} by {artist1} and {track2} by {artist2} give you the clearest starting points, while the later picks branch into {safe_style_1} and {safe_style_2}.

This is where compact_broad is acceptable, because broad turns naturally need concise variety. Still keep it grounded.

E. Profile/history-aware response

Only use when you have reliable profile/history evidence.

Because your profile/history points toward {known_preference}, I prioritized tracks with {matching_feature}. {track1} by {artist1} fits that best, and {track2} by {artist2} gives a slightly different angle without leaving the same lane.

5. Anti-template rules

These are more important than adding more templates.

Ban generic opener repetition

Do not overuse:

Here are some recommendations...
I think you’ll enjoy...
Based on your preferences...
These tracks should fit...

Use varied but grounded openers:

I kept this close to {entity}.
I leaned into the {mood} side of your request.
I adjusted away from {negative_feedback}.
For this turn, I treated {constraint} as the main signal.
This list starts with safer matches, then branches out slightly.
Ban explanation without evidence

Bad:

These songs match your vibe perfectly.

Good:

These lean into the mellow electronic tags you asked for, with the first two kept closest to the seed artist.
Ban fake personalization

Bad:

Since you love emotional songs...

Good:

Since this turn asks for something more emotional...

or:

Since earlier feedback moved away from high-energy picks...
Ban excessive title dumping

Bad:

Try A, B, C, D, E, F, and G.

Good:

Mention only 2-3 tracks/artists, then explain the list shape.
Ban mechanical diversity language

Avoid:

I included diverse tracks to maximize variety.
This list balances relevance and diversity.
These recommendations optimize your listening experience.

Use:

I kept the first few close to the request, then used the later spots for nearby discoveries.
6. Controlled lexical diversity without mechanical writing

Distinct-2 should come from real contextual variation, not random synonyms.

Use rotating phrase banks for these components only:

Acknowledgement bank

I leaned into...
I treated this as...
I adjusted the list toward...
I used your current request as the main signal for...
Explanation connector bank
because

which makes it a good fit for
so it works as
while
then
to keep the list from getting too narrow
List-shape bank
starts with the safest matches
branches into nearby discoveries
keeps the same mood without repeating one lane
stays close to the seed before widening out
mixes direct matches with a few softer alternatives

Do not rotate entire sentence skeletons too aggressively. That creates compact_broad-style mechanical variety.

Good diversity source:

Different category-specific reasoning.
Different metadata slots.
Different current-turn hooks.
Different feedback hooks.
Different list-shape explanations.

Bad diversity source:

Random synonym swaps.
Long rare phrases.
Unnatural adjectives.
Changing wording while saying the same generic thing.

8. Exact implementation checklist

For every generated response, run a self-check:

1. Does it mention at least one top-20 track title or artist?
2. Does it mention the current user request or one extracted constraint?
3. Does it include one grounded reason from clean metadata?
4. If feedback exists, does it acknowledge the direction of the feedback?
5. If profile/history exists, does it use it cautiously?
6. Does it avoid banned noisy tags?
7. Does it avoid unverified audio/lyric claims?
8. Is length between 35 and 70 words?
9. Is it not identical in skeleton to the previous N responses?
10. Does it avoid saying “diversity”, “optimize”, “ranking”, or “metadata”?

If any of 1-7 fails, regenerate with safe fallback.

Safe fallback:

I kept the recommendations close to your current request and used the first few tracks as the strongest matches. Start with {track1} by {artist1}; {track2} by {artist2} follows the same {clean_genre_or_mood} direction. The later picks stay nearby while giving you a little room to discover something new.
9. Risk of over-optimizing Distinct-2

Distinct-2 can be gamed by making responses lexically varied, but that does not guarantee the response is helpful. In the official setup, Lexical Diversity is only one response metric, while LLM-as-a-Judge is also evaluated.

Risks:

1. Mechanical variety
   The model uses many surface forms but feels templated.

2. Weak personalization
   Responses sound different but do not use the user’s actual turn/history.

3. Noisy metadata leakage
   Rare tags boost bigram diversity but look absurd to the judge.

4. Hallucination pressure
   Trying to sound rich causes fake audio/lyric claims.

5. Overlong responses
   More words increase bigram diversity but may dilute the recommendation.

6. Style mismatch
   Music assistant should sound clear and useful, not like a benchmark optimizer.

Your current numbers suggest compact_broad may be near that danger zone: excellent Distinct-2, but weaker perceived naturalness. Since prior public judge was 1.0, I would not sacrifice judge readability for another +0.09 Blind-A Distinct-2.
